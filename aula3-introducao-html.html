Abertura da aula

Antes de instalar qualquer coisa, é importante situar a turma: o que vem por aí, quanto tempo isso deve levar e por que isso importa.

professora fala
— Bom dia, turma! Terceira aula, e hoje é um dia especial.

Até agora a gente conversou sobre lógica, sobre como um computador "pensa" e sobre o caminho que vocês vão trilhar nesse curso. Hoje a gente sai da teoria e coloca a mão na massa pela primeira vez de verdade.

01

Preparando a caixa de ferramentas

Toda profissão tem suas ferramentas. Pedreiro tem colher de pedreiro, cozinheiro tem faca boa e afiada. A nossa caixa de ferramentas de hoje: o Git, uma conta no GitHub e o VSCode configurado.

professora fala
— Pensem assim, com calma, porque isso vai acompanhar vocês o curso inteiro:

O VSCode é a nossa "folha de papel e caneta" — é onde a gente vai escrever o código, todos os dias, em todas as aulas daqui pra frente.

O Git é diferente: ele não é onde a gente escreve código, é uma ferramenta que fica "por trás", rodando no computador de vocês, guardando um histórico de tudo que foi escrito. Pensem nele como uma máquina do tempo, ou como o "Ctrl+Z" definitivo: se em algum momento vocês estragarem um projeto inteiro sem querer, o Git guarda versões anteriores para vocês voltarem no tempo.

O GitHub, por sua vez, é um site — uma espécie de "Google Drive" só que feito especialmente para guardar código na nuvem. Ele usa o Git por trás dos panos, mas o GitHub em si é o lugar onde o código de vocês fica hospedado, visível, e onde vocês vão poder mostrar seus projetos pro mundo (incluindo recrutadores, no futuro).

Repare que são três coisas diferentes, mas que trabalham juntas: VSCode (onde escrevo), Git (que guarda o histórico local) e GitHub (onde guardo esse histórico na nuvem, para todo mundo ver).

respire
Hoje a gente só vai instalar essas ferramentas e criar a conta. O uso de verdade do Git — salvando versões, subindo pro GitHub — a gente aprende com calma numa aula futura dedicada só a isso. Por hoje, é só deixar tudo pronto na "bancada de trabalho".

1.1 Instalando o Git

mostrar na tela
Compartilhar tela e abrir o site oficial git-scm.com junto com a turma, mostrando cada clique.
  1. Acessem git-scm.com no navegador. É o site oficial — desconfiem de qualquer outro site que prometa "baixar Git" com botões piscando ou muita propaganda. Dica: digitem o endereço direto na barra do navegador, em vez de pesquisar no Google, para não cair em um site falso.
  2. Cliquem em Download. O site detecta sozinho se vocês usam Windows, Mac ou Linux e já sugere a versão certa.
  3. Abram o instalador baixado. Durante a instalação, se aparecer uma tela de opções e vocês não tiverem certeza do que marcar, deixem tudo como está por padrão e cliquem em "Next" até o fim — os padrões já servem perfeitamente pra gente agora. Não precisam entender cada opção nesse momento. Se aparecer uma pergunta sobre "editor padrão", pode deixar como Vim ou o que vier marcado — a gente vai usar o VSCode de qualquer forma, isso não afeta nosso curso.
  4. Para confirmar que funcionou, precisamos abrir um terminal: no Windows, procurem no menu iniciar por "Git Bash" (ele já vem instalado junto com o Git); no Mac ou Linux, abram o aplicativo "Terminal".
  5. Dentro do terminal, digitem git --version e apertem Enter. Se aparecer um número de versão (algo como git version 2.44.0), deu tudo certo!
pergunta comum
"O que é esse terminal, professora?" — É uma tela onde a gente conversa com o computador escrevendo comandos de texto, em vez de clicar em ícones e menus. Parece assustador na primeira vez — uma telinha preta com letrinhas — mas em poucas aulas vai virar rotina, e vocês vão perceber que é até mais rápido que usar o mouse para muita coisa.

1.2 Criando a conta no GitHub

professora fala

Vamos criar a conta agora. Pensem nela como o "perfil profissional" de vocês como programadores — muitas empresas, na hora de contratar, dão uma olhada no GitHub da pessoa candidata para ver os projetos que ela já fez.

  1. Acessem github.com e cliquem em Sign up (Cadastrar-se), geralmente no canto superior direito da página.
  2. Preencham e-mail e senha quando solicitado. Usem um e-mail que vocês acessam com frequência, porque o GitHub vai enviar um código de confirmação para ele.
  3. Escolham um nome de usuário profissional — pensem que, com o tempo, esse é o endereço que vocês vão colocar no currículo e no LinkedIn (algo como github.com/seu-usuario). Evitem apelidos de adolescência ou combinações de números aleatórios aqui! Boas opções costumam ser variações do próprio nome, como "mariasilva-dev" ou "joaocosta-tech".
  4. Confirmem o e-mail quando o GitHub pedir, digitando o código que chegou na caixa de entrada — sem esse passo a conta fica incompleta e algumas funções não funcionam.
  5. Prontinho: essa conta vai ser a "casa na nuvem" de todos os projetos que vocês criarem no curso, e a gente vai voltar a usá-la em detalhes numa aula futura.

1.3 Instalando o VSCode

professora fala
— O VSCode vai ser a casa de vocês pelos próximos meses. Vamos conhecer ele com calma.

VSCode é a abreviação de Visual Studio Code, um editor de código gratuito, feito pela Microsoft, usado por profissionais no mundo inteiro — de estagiários a engenheiros sêniores em grandes empresas de tecnologia. Ele é meio "cru" recém-instalado, parecido com um caderno em branco, mas a gente vai deixar ele mais bonito e útil com algumas extensões daqui a pouco.

  1. Acessem code.visualstudio.com e cliquem no botão de download para o sistema operacional de vocês (o site também costuma detectar automaticamente).
  2. Instalem normalmente, sempre com as opções padrão marcadas. No Windows, se aparecerem caixinhas de seleção extras como "Adicionar ação 'Abrir com Code' ao menu de contexto", pode deixar marcadas — isso só facilita a vida de vocês depois.
  3. Abram o VSCode pela primeira vez. Vão ver uma tela de boas-vindas com dicas — podem fechar essa aba por enquanto, ou deixar aberta para explorar depois.
  4. Vamos fazer um tour rápido pela interface: no canto esquerdo, aquela barra fina de ícones verticais é a Activity Bar; o painel que abre ao lado dela (normalmente mostrando arquivos) é o Explorer; a área grande no meio é onde o código aparece; e lá embaixo fica a barra de status, mostrando informações como o tipo de arquivo aberto.
  5. No menu lateral esquerdo, procurem o ícone de quadradinhos empilhados (Extensões) e cliquem nele — é ali que vamos instalar nossas ferramentas extras.
mostrar na tela
Instalar ao vivo, junto com a turma, cada uma destas extensões (buscar pelo nome exato na barra de busca de extensões, clicar em "Install" e aguardar carregar):
  • ✓Portuguese Language Pack — traduz a interface do VSCode para português. É opcional: muitos programadores preferem deixar em inglês desde cedo, já que a maior parte da documentação e das mensagens de erro do mundo todo vem em inglês. Fica a escolha de vocês.
  • ✓Live Server — abre a nossa página HTML diretamente no navegador e atualiza sozinha toda vez que a gente salva o arquivo, sem precisar apertar F5 manualmente. Essa vai ser usada em praticamente toda aula prática.
  • ✓Prettier - Code formatter — organiza e "arruma" a identação e a formatação do nosso código automaticamente, deixando tudo alinhado e legível com um atalho de teclado.
  • ✓Auto Rename Tag — quando editamos o nome de uma tag de abertura em HTML, ela renomeia a de fechamento automaticamente junto. Uma mãozinha e tanto pra evitar erros bobos!
  • ✓Material Icon Theme — deixa os ícones de pastas e arquivos mais bonitos e, principalmente, mais fáceis de identificar rapidamente pelo tipo de arquivo.
pergunta comum
"Preciso reiniciar o VSCode depois de instalar uma extensão?" — Na maioria das vezes não, mas se alguma extensão não parecer estar funcionando (como o Live Server não aparecendo no menu de botão direito), feche e abra o VSCode novamente. Isso resolve praticamente sempre.
respire
Ufa. Instalação feita! Deem uma esticada, bebam uma água, olhem pra longe por 20 segundos para descansar os olhos. Quando todo mundo tiver o Git, o GitHub e o VSCode com as extensões prontos, a gente segue para a parte mais gostosa da aula: entender o que diabos é esse tal de HTML.
</instalação>
02

O que é HTML, afinal?

Vamos construir essa ideia com calma, camada por camada, antes de ver qualquer código de verdade.

professora fala
— Deixa eu te fazer uma pergunta: quando vocês entram num site, o que vocês estão vendo?

Título, parágrafo de texto, uma imagem ali, um botão aqui, um menu no topo... Pois é. Tudo isso que aparece na tela precisa ser organizado de alguma forma para o navegador entender o que é o quê. O navegador — Chrome, Firefox, Safari, qualquer um — não sabe, por padrão, o que é um título e o que é uma imagem só de olhar um monte de texto solto. Alguém precisa dizer isso pra ele, de um jeito que ele entenda.

É exatamente para isso que o HTML existe. HTML significa HyperText Markup Language — em português, "Linguagem de Marcação de Hipertexto". O nome é feio e confuso, então vamos separar em pedaços:

  • HyperText (Hipertexto): é um texto que pode conter links — referências que levam para outros textos, outras páginas. É basicamente o que torna a web uma "teia" (web = teia): páginas conectadas umas às outras por links.
  • Markup (Marcação): é o ato de "marcar" um texto, como quando a gente grifa um trecho de um livro com marca-texto para destacar que aquilo é importante. Só que aqui, em vez de simplesmente destacar, a gente está dizendo o tipo daquele conteúdo.
  • Language (Linguagem): porque, assim como o português ou o inglês têm regras de gramática, o HTML também tem suas próprias regras — que vamos aprender juntos hoje.

Juntando tudo: HTML é a linguagem que a gente usa para marcar o conteúdo de uma página, dizendo ao navegador: "isso aqui é um título", "isso aqui é um parágrafo", "isso aqui é uma imagem", "isso aqui é um link para outra página".

professora fala
— Agora, a analogia que eu quero que vocês guardem para o resto do curso, porque ela vai se repetir toda aula:

HTML é o esqueleto de uma página.

Pensem no corpo humano. O esqueleto não é o que a gente vê quando olha pra uma pessoa na rua — quem "aparece" é a pele, a roupa, o cabelo, a expressão do rosto. Mas sem o esqueleto por baixo de tudo isso, o corpo não tem forma nenhuma, não fica em pé, não tem onde pendurar os músculos e os órgãos.

O HTML é exatamente isso para um site: ele dá estrutura, o "arcabouço" sobre o qual tudo o mais é construído. Ele decide "aqui vai a cabeça (o título principal), aqui vai o tronco (o conteúdo principal da página), aqui vai um braço (uma imagem), aqui vai uma mão apontando para outro lugar (um link)".

Mais pra frente no curso, quando aprendermos CSS, a gente vai "vestir" esse esqueleto — dar cor, forma, espaçamento, um visual bonito. E depois, com JavaScript, a gente vai fazer esse corpo se mexer — reagir a cliques, mostrar e esconder coisas, calcular valores. Mas hoje é só sobre o esqueleto: sem ele, não existe corpo nenhum para vestir ou fazer se mexer.

professora fala
— E de onde vem o HTML? Um contexto rápido, sem precisar decorar datas:

O HTML foi criado no início dos anos 1990 por um cientista chamado Tim Berners-Lee, junto com a própria ideia da World Wide Web (a "www" que a gente ainda vê em alguns endereços). A ideia original era simples: permitir que pesquisadores compartilhassem documentos de texto conectados por links, de qualquer lugar do mundo. Só depois, com o passar dos anos, o HTML foi ganhando mais recursos — imagens, formulários, vídeos, e assim por diante — até chegar na web riquíssima que a gente usa hoje.

O motivo de eu contar isso é simples: o HTML não foi inventado para ser "bonito" — ele foi inventado para estruturar informação de forma que qualquer computador do mundo consiga entender. Beleza visual é trabalho de outra ferramenta, que veremos mais pra frente.

deixa entrar
Repitam isso mentalmente, porque vai ser cobrado o curso inteiro: HTML = estrutura. CSS = estilo. JavaScript = comportamento. Essa frase vai fazer todo sentido daqui a algumas aulas — por enquanto, guardem principalmente a primeira parte, é sobre ela que vamos trabalhar hoje.
</o-que-é-html>
03

Como o esqueleto é montado: tags, elementos e atributos

Três palavras que vocês vão ouvir o curso inteiro. Vamos entender cada uma com calma, uma peça de cada vez, e ainda ver algumas regras importantes de escrita.

professora fala
— Vamos começar pela peça mais básica: a tag.

Uma tag é uma etiqueta, escrita entre os sinais de menor < e maior >, que diz ao navegador o tipo daquele pedaço de conteúdo. Quase toda tag em HTML vem em par: uma de abertura e uma de fechamento — e a única diferença entre elas é a barrinha / logo depois do < na de fechamento.

exemplo.html
<p>Esse é um parágrafo de texto.</p>
professora fala
— Quando uma tag de abertura, o conteúdo de dentro e a tag de fechamento aparecem juntos, damos um nome pra esse "pacote completo":

chamamos isso de elemento. Ou seja: a tag é a etiqueta, e o elemento é o "pacote inteiro" — etiqueta de abertura, conteúdo, etiqueta de fechamento, tudo junto.

Pensem de novo no esqueleto: a tag <p> é como uma placa dizendo "aqui começa um osso do tipo parágrafo", e o elemento inteiro é o osso montado, do início ao fim, já com sua "carne" (o conteúdo) dentro.

Uma coisa importante: elementos podem ficar dentro de outros elementos — isso se chama aninhamento (em inglês, nesting). É como caixas dentro de caixas. Olhem esse exemplo:

exemplo.html
<p>Esse texto tem uma palavra <strong>importante</strong> no meio.</p>
professora fala

Reparem: o elemento <strong> (que deixa o texto em negrito) está dentro do elemento <p>. Isso é perfeitamente normal e vai acontecer o tempo todo. A única regra de ouro aqui é: quem abre por último, fecha primeiro — como quando a gente abre parênteses dentro de parênteses. Se o <strong> abriu depois do <p>, ele precisa fechar antes do </p>. Nunca cruzado.

certo-vs-errado.html
// certo: o strong fecha antes do p
<p>texto <strong>forte</strong></p>

// errado: as tags se cruzam
<p>texto <strong>forte</p></strong>
professora fala
— E os atributos?

Às vezes, a tag sozinha não dá conta de toda informação que a gente precisa passar. Por exemplo: se eu quero mostrar uma imagem, o navegador precisa saber qual imagem — o nome da tag <img> sozinho não diz isso. Para complementar, usamos atributos: informações extras que colocamos dentro da tag de abertura, sempre no formato nome="valor".

exemplo.html
<img src="gato.jpg" alt="Foto de um gato laranja dormindo">
professora fala

Reparem: src (abreviação de source, "fonte" em inglês) é o atributo que diz onde está o arquivo da imagem, e alt (de alternative, "alternativo") é o atributo que diz o que tem nessa imagem, caso ela não carregue por algum motivo — ou caso a pessoa esteja usando um leitor de tela por ter dificuldade de enxergar. Esse segundo atributo é importante não só por acessibilidade, mas porque o Google e outros buscadores usam esse texto para entender do que se trata a imagem.

Reparem também que a tag <img> não tem uma tag de fechamento separada — ela é o que chamamos de elemento vazio (ou void element), porque uma imagem não "envolve" nenhum conteúdo dentro dela, então não faz sentido ter um </img>. Outros elementos vazios comuns são o <br> (quebra de linha) e o <hr> (linha horizontal divisória) — vamos ver os dois em detalhe daqui a pouco.

3.1 Mais algumas regras importantes de escrita

professora fala
  • Espaços em branco e quebras de linha não importam para o HTML. O navegador junta tudo e ignora espaços extras, tabulações e linhas em branco entre as tags. Ou seja: a identação que a gente usa (aqueles espaços no início da linha, "escadinha") não é obrigatória para o código funcionar — ela existe só para nós, humanos, conseguirmos ler o código com mais facilidade. Mesmo assim, é uma prática essencial: código bem identado é muito mais fácil de entender e de corrigir.
  • Tags em HTML não diferenciam maiúsculas de minúsculas — <P> e <p> funcionam do mesmo jeito. Mas o combinado do mercado, e o que vamos seguir aqui, é sempre escrever tudo em minúsculas. Isso é só uma convenção de organização, mas seguir convenções é parte importante de trabalhar em equipe.
  • Comentários são trechos de texto que a gente escreve no código só para anotações nossas — o navegador simplesmente os ignora, eles nunca aparecem na tela. Em HTML, escrevemos assim: <!-- isso é um comentário -->. São úteis para deixar lembretes ou desativar temporariamente um trecho de código sem apagá-lo.
comentario.html
<!-- Essa é a seção de apresentação da página -->
<h1>Olá, mundo!</h1>
respire
Recapitulando com uma frase só: tag é a etiqueta, elemento é o pacote completo (abertura + conteúdo + fechamento), e atributo é uma informação extra que a gente coloca dentro da tag de abertura. Elementos podem ficar aninhados uns dentro dos outros, sempre respeitando a ordem de fechamento. Se isso ainda parece nebuloso, tudo bem — daqui a pouco a gente vê isso rodando de verdade no navegador, e costuma clicar na hora.
</tags-elementos-atributos>
04

Conhecendo os primeiros "ossos"

Não precisamos decorar todas as tags que existem — são dezenas. Hoje, vamos conhecer bem o essencial para montar nossa primeira página com confiança.

professora fala
— Antes de qualquer tag de conteúdo, toda página HTML precisa de um "molde" básico.

É tipo a folha de papel dobrada de um jeito específico antes de começar a escrever uma carta formal. Vamos ver essa estrutura completa daqui a pouco, no nosso Hello World. Por enquanto, vamos conhecer bem as tags de conteúdo que vamos usar hoje — e algumas extras, para quando vocês forem praticar em casa.

<h1> a <h6>

Títulos e subtítulos, do maior (h1) para o menor (h6). Pensem como capítulo, seção, subseção. Toda página deve ter só um h1, que é o título principal.

<p>

Um parágrafo de texto comum. É a tag que vocês mais vão usar no dia a dia para blocos de texto corrido.

<img>

Exibe uma imagem. Sempre acompanhada dos atributos src (de onde vem a imagem) e alt (texto alternativo). Não tem tag de fechamento.

<a>

Um link (âncora). Usa o atributo href para dizer para onde o link leva quando clicado. O texto entre as tags é o que fica clicável.

<strong>

Deixa o texto em negrito e indica que aquele trecho tem importância semântica — não é só um efeito visual, é um significado.

<em>

Deixa o texto em itálico, indicando ênfase na fala — como quando a gente grifa a voz ao falar uma palavra com mais força.

<br>

Quebra de linha simples, sem começar um novo parágrafo. Usar com moderação — geralmente parágrafos separados são a escolha melhor.

<hr>

Desenha uma linha horizontal, útil para separar visualmente blocos de conteúdo diferentes na página.

<div> e <span>

Duas caixas "neutras", sem significado próprio, usadas para agrupar outros elementos. div agrupa blocos maiores; span agrupa pedacinhos de texto. Vamos usar bastante nas próximas aulas, junto com o CSS.

<ul>, <ol>, <li>

Listas! ul é uma lista sem numeração (com marcadores), ol é uma lista numerada, e cada item de qualquer uma das duas é um li.

mostrar no navegador
Digitar ao vivo, na tela, cada uma dessas tags dentro de um arquivo de teste e salvar com o Live Server ligado, para a turma ver o resultado aparecendo no navegador em tempo real, uma tag de cada vez, comentando a aparência padrão de cada uma.
teste.html
<h1>Bem-vindos ao meu site</h1>
<p>Esse é o meu primeiro parágrafo em HTML, e essa parte é <strong>bem importante</strong>.</p>
<img src="foto.jpg" alt="Uma paisagem de montanhas">
<a href="https://google.com">Clique aqui para pesquisar</a>
<ul>
    <li>Primeiro item</li>
    <li>Segundo item</li>
</ul>
professora fala

Vejam como cada tag "avisa" o navegador do que se trata, e o navegador automaticamente já sabe como mostrar aquilo: título grande e em negrito, parágrafo com texto normal, imagem exibida na tela, link sublinhado e azul, lista com bolinhas na frente de cada item. A gente não precisou dizer como deveria parecer, nem escolher cor ou tamanho de fonte — isso é trabalho do CSS, mais pra frente. O HTML só diz o que é cada coisa, e o navegador já vem com uma aparência padrão para cada tipo de elemento.

pergunta comum
"Professora, e se eu esquecer de fechar uma tag?" — O navegador é bem tolerante e, na maioria das vezes, ele vai tentar "adivinhar" e mostrar algo mesmo assim. Mas isso pode bagunçar a página de formas estranhas, difíceis de perceber. Por isso o hábito certo, desde já, é: abriu uma tag, já escreve a de fechamento na mesma hora, antes de preencher o conteúdo no meio. O VSCode, inclusive, ajuda muito nisso — vamos ver como.
</tags-essenciais>
05

Mãos à obra: o primeiro Hello World

Chegou a hora. Vamos, juntos, criar o primeiro arquivo HTML de vocês, do zero, dentro do VSCode — sem pular nenhum passo.

professora fala
— Bora criar de verdade. Sigam comigo, passo a passo, sem pressa. Quem terminar antes, espera a turma.
  1. Criem, em algum lugar do computador, uma pasta chamada meu-primeiro-site. Pode ser na área de trabalho mesmo — depois vamos organizar melhor isso com o Git. Evitem espaços, acentos ou caracteres especiais no nome da pasta — prefiram hífen ou underline para separar palavras. Isso evita bugs chatos mais pra frente.
  2. Abram o VSCode e, no menu File > Open Folder (ou Arquivo > Abrir Pasta), selecionem essa pasta que acabaram de criar.
  3. No painel do lado esquerdo (o Explorer), cliquem no ícone de "novo arquivo" e nomeiem o arquivo exatamente como index.html — esse nome não é por acaso: é o nome padrão que os navegadores e servidores procuram primeiro quando abrem um site, sem que a gente precise digitar o nome do arquivo na URL.
  4. Dentro do arquivo vazio, digitem apenas o caractere ! e apertem a tecla Tab. O VSCode vai gerar sozinho toda a estrutura básica do HTML — essa mágica se chama Emmet, um recurso que já vem embutido no editor, sem precisar instalar nada.
mostrar na tela

professora fala
— Vamos entender rapidamente cada uma dessas peças, com calma, sem susto:

<!DOCTYPE html> é só um aviso lá no topo, sempre na primeira linha, avisando ao navegador "ei, esse arquivo é um HTML moderno, seguindo o padrão atual". É a única linha do arquivo que não é uma tag de verdade — é uma declaração especial.

<html lang="pt-BR"> é a "casca" que envolve a página inteira — é literalmente o esqueleto todo, da cabeça aos pés, tudo mais fica dentro dela. O atributo lang diz em qual idioma a página está escrita, o que ajuda leitores de tela e buscadores a entenderem o conteúdo corretamente.

Dentro do <html> temos duas partes bem separadas: o <head> e o <body>.

  • <head> guarda informações sobre a página que não aparecem diretamente na tela para quem visita o site. Dentro dele temos: o <meta charset="UTF-8">, que garante que acentos e caracteres especiais do português (ç, ã, é) apareçam corretamente; o <meta name="viewport"...>, que ajuda a página a se adaptar bem a telas de celular; e o <title>, que define o texto que aparece na aba do navegador.
  • <body> é onde vai todo o conteúdo que aparece de verdade na tela — títulos, textos, imagens, links, listas, tudo o que vimos na seção anterior entra aqui dentro.
pergunta comum
"Preciso decorar essa estrutura toda?" — Não! Por isso usamos o atalho ! + Tab — ele existe justamente para gerar essa estrutura automaticamente, sempre. O importante hoje é entender o papel de cada parte, não memorizar a sintaxe exata.
  1. Cliquem dentro das tags <body></body> e alterem o <title> lá no <head> para algo como Meu Primeiro Site — isso é o que vai aparecer na aba do navegador.
  2. Agora escrevam o nosso "Hello World" dentro do <body>:
index.html
<body>
    <h1>Olá, mundo!</h1>
    <p>Essa é a minha primeira página em HTML.</p>
</body>
  1. Salvem o arquivo com Ctrl + S (Windows/Linux) ou Cmd + S (Mac). Esse é, provavelmente, o atalho que vocês mais vão usar no curso inteiro — criem o hábito de salvar com frequência.
  2. Cliquem com o botão direito em qualquer parte do arquivo e escolham Open with Live Server. O navegador vai abrir sozinho, numa nova aba, mostrando a página de vocês rodando de verdade, com um endereço parecido com 127.0.0.1:5500. Esse endereço estranho é o "endereço local" do computador de vocês — significa que a página está rodando só na máquina de vocês, ainda não está publicada na internet para o mundo ver. Isso é normal e é assim que trabalhamos durante todo o desenvolvimento.
  3. Agora o teste final, pra sentir a mágica do Live Server: voltem ao VSCode, mudem o texto do <h1> para o nome de vocês, salvem de novo com Ctrl+S, e observem o navegador atualizar sozinho, sem vocês precisarem clicar em nada nele.
momento importante
Parem um segundo aqui. Olhem pra essa tela. Isso que está aparecendo no navegador — por mais simples que pareça — foi escrito por vocês, do zero, entendendo cada linha, sem copiar e colar de lugar nenhum. Essa é a página número um de muitas, muitas outras que vocês ainda vão construir. Guardem essa sensação, porque ela só vai crescer.

06

Perguntas frequentes

Uma coletânea das dúvidas que mais aparecem nessa fase do curso. Cliquem em cada pergunta para abrir a resposta — guardem esse material, vocês vão voltar nele mais de uma vez.

P Preciso ter internet o tempo todo para escrever HTML?▾
Não. Depois de instalado, o VSCode funciona totalmente offline para escrever e visualizar suas páginas HTML no navegador com o Live Server. Você só precisa de internet para instalar as ferramentas, buscar imagens, ou quando começar a usar o GitHub para guardar o código na nuvem.
P Qual a diferença entre HTML e um site "pronto", tipo os feitos no Canva ou Wix?▾
Ferramentas como Wix e Canva geram o HTML por trás dos panos, automaticamente, sem você ver ou controlar o código. Aprender HTML "na mão" te dá controle total sobre a estrutura, entendimento de como a web funciona por dentro, e é a base para qualquer caminho profissional em desenvolvimento — mesmo quem usa essas ferramentas no dia a dia se beneficia de entender o que está rolando por baixo.
P Posso escrever HTML no Bloco de Notas em vez do VSCode?▾
Tecnicamente sim, HTML é só texto puro. Mas o VSCode oferece destaque de cores por tipo de código (chamado syntax highlighting), autocompletar, detecção de erros, o atalho do Emmet (!+Tab) e a extensão Live Server — recursos que tornam o aprendizado muito mais rápido e com menos erros bobos. Não há motivo para abrir mão disso.
P O que acontece se eu usar uma tag que não existe, tipo <bananinha>?▾
O navegador não trava nem mostra um erro chamativo — ele simplesmente ignora que aquilo é uma tag especial e trata como se fosse um elemento genérico, sem nenhum estilo padrão. Ou seja: nada quebra visivelmente, mas também nada funciona como deveria. Por isso é importante usar sempre as tags reais da linguagem.
P Toda tag precisa ter um atributo?▾
Não! A maioria das tags que vimos hoje (h1, p, strong, ul, li) funciona perfeitamente sem nenhum atributo. Atributos só entram quando a tag precisa de uma informação extra para funcionar — como o src do <img> ou o href do <a>.
P Por que às vezes vejo HTML escrito todo em uma linha só, sem identação?▾
Isso costuma acontecer com código gerado automaticamente por ferramentas, ou "minificado" (comprimido) para deixar o arquivo mais leve para carregar na internet. Para nós, aprendendo, sempre vamos escrever identado e organizado — é assim que se escreve HTML "à mão", pensando em quem vai ler o código depois, inclusive vocês mesmos no futuro.
P Existe limite de quantas tags eu posso usar numa página?▾
Não, nenhum limite prático. Sites reais têm páginas com centenas ou milhares de elementos HTML. O que existe é uma preocupação crescente, conforme o curso avança, com organização — para o código não virar uma bagunça difícil de entender e manter.
P Git, GitHub e GitHub Desktop são a mesma coisa?▾
Não. Git é a ferramenta de controle de versão em si, que roda no seu computador pelo terminal. GitHub é o site que hospeda repositórios na nuvem. GitHub Desktop é um programa opcional, com interface visual (botões e cliques), que facilita usar o Git sem digitar comandos no terminal. Vamos explorar essas diferenças com calma numa aula futura dedicada ao Git.
</perguntas-frequentes>
07

Glossário da aula

Todos os termos novos de hoje, reunidos num só lugar, em ordem alfabética, para consulta rápida sempre que precisarem.

Atributo

Informação extra colocada dentro de uma tag de abertura, no formato nome="valor", como o src de uma imagem.

Aninhamento (nesting)

Quando um elemento HTML fica escrito dentro de outro elemento, formando camadas, como caixas dentro de caixas.

<body>

Parte do documento HTML que contém todo o conteúdo visível na tela do navegador.

Comentário

Trecho de código, escrito como <!-- texto -->, que o navegador ignora e nunca exibe. Serve para anotações do próprio programador.

DOCTYPE

Declaração no topo do arquivo HTML que informa ao navegador qual versão do HTML está sendo usada.

Editor de código

Programa usado para escrever código, com recursos como destaque de cores e autocompletar. Nosso editor é o VSCode.

Elemento

O "pacote completo" formado pela tag de abertura, o conteúdo dentro dela, e a tag de fechamento.

Elemento vazio (void element)

Tag que não possui conteúdo interno nem tag de fechamento própria, como <img> ou <br>.

Emmet

Recurso já embutido no VSCode que expande atalhos curtos, como ! + Tab, em blocos inteiros de código HTML.

Extensão (VSCode)

Complemento instalável que adiciona novas funções ao editor, como o Live Server ou o Prettier.

Git

Ferramenta de controle de versão que guarda um histórico das mudanças feitas nos arquivos de um projeto ao longo do tempo.

GitHub

Site que hospeda repositórios de código na nuvem, usando o Git por trás, permitindo compartilhar e colaborar em projetos.

<head>

Parte do documento HTML que guarda informações sobre a página que não aparecem diretamente na tela.

HTML

Sigla de HyperText Markup Language — a linguagem usada para estruturar e marcar o conteúdo de páginas web.

Identação

Espaços no início da linha usados para organizar visualmente o código e mostrar o nível de aninhamento dos elementos.

Live Server

Extensão do VSCode que abre a página HTML no navegador e a atualiza automaticamente a cada vez que o arquivo é salvo.

Navegador

Programa usado para acessar e exibir páginas da web, como Chrome, Firefox, Safari ou Edge.

Tag

Etiqueta escrita entre os sinais < e > que indica ao navegador o tipo de um conteúdo.

Terminal

Tela onde comandos de texto são digitados para conversar diretamente com o computador, sem usar cliques em ícones.

VSCode

Abreviação de Visual Studio Code, o editor de código gratuito que vamos usar durante todo o curso.

</glossário>
