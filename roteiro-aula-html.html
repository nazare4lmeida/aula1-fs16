01

Relembrando: o que é HTML?

⏱️ ~10 min

O que você vai fazer neste bloco: reativar a memória da aula passada em 10 minutos. A revisão vem ANTES da instalação de propósito: relembrar o "porquê" antes do "como" faz você chegar na parte técnica sabendo para que serve cada ferramenta.

A aula passada foi teoria: o que é HTML, tags, atributos... Hoje é o grande dia de sujar as mãos — e do jeito completo: você vai montar seu ambiente de desenvolvimento DO ZERO — editor, Git, conta no GitHub — e, com tudo pronto, escrever sua primeira página de verdade. No fim, você terá o computador de um dev e um site funcionando nele.

Antes de seguir, teste rápido de memória — sem medo de errar: você lembra o que significa a sigla HTML? (Resposta: HyperText Markup Language — linguagem de marcação de hipertexto.)

Recapitulação em 3 frases (a analogia da casa)

Lembra da casa? Um site é como uma casa: o HTML é a estrutura — paredes, portas, cômodos. O CSS, que vem nas próximas aulas, é a decoração: cor da parede, móveis, estilo. E o JavaScript é a parte elétrica e o encanamento: o que faz a casa funcionar — luz acende, campainha toca.

Hoje você constrói só a estrutura. E olha: uma casa sem pintura é feia, mas fica de pé. Uma casa sem estrutura... nem existe. Por isso HTML vem primeiro.

Ponto importante para guardar: HTML não é linguagem de programação — é linguagem de marcação. Ela não toma decisões, não faz contas. Ela descreve e organiza conteúdo: "isto é um título", "isto é um parágrafo", "isto é uma tabela".

Pense num texto no Word onde você seleciona algo e marca como "Título 1". O HTML faz isso, só que escrevendo a marcação à mão. Você não está programando lógica ainda — está etiquetando conteúdo para o navegador entender.

O que você vai construir

🖥️ Conceito

O projeto se chama "Bem-cuidado": um organizador de consultas médicas para a Dona Lúcia — com próximas consultas, histórico, exames anexados e um formulário de lembretes.

Pensa naquela avó ou mãe que vive perdendo o papelzinho da consulta — é para ela que você vai construir. Conectar o projeto com a vida real deixa tudo mais fácil de aprender.

02

Revisão-relâmpago: tags, elementos e atributos

⏱️ ~15 min

O que você vai fazer neste bloco: revisar os 4 conceitos-chave da aula passada. Tente responder mentalmente cada pergunta antes de ler a explicação — funciona muito melhor que só ler.

2.1 — Tag: a etiqueta

Tag, em inglês, é etiqueta. É literalmente isso: uma etiqueta que você cola no conteúdo para dizer o que ele é. Toda tag vem entre os sinais de menor e maior: <p>, <h1>, <table>.

E quase toda tag anda em dupla: a que abre e a que fecha. A que fecha tem uma barra: </p>. É como abrir e fechar aspas num texto — se você abre e não fecha, vira bagunça.

🖥️ Conceito

Observe a anatomia de uma tag:

<p>Você tem 2 consultas nesta semana.</p>
↑ abre     ↑ conteúdo                    ↑ fecha

2.2 — Elemento: o pacote completo

Elemento = tag de abertura + conteúdo + tag de fechamento. A tag é a etiqueta; o elemento é a caixa inteira com a etiqueta colada.

Uma exceção para guardar: algumas tags não têm conteúdo dentro, então não precisam fechar. São as tags vazias, como <br /> (quebra de linha), <input /> (campo de formulário) e <meta />. Elas são autossuficientes — pense numa etiqueta que já é o produto inteiro.

2.3 — Atributo: a informação extra

Atributo é uma informação extra que escrevemos dentro da tag de abertura, sempre no formato nome="valor". Pense assim: a tag diz o que o elemento é; o atributo diz como ele é ou detalhes sobre ele.

Exemplo do mundo real: "camiseta" é a tag. Tamanho M, cor azul — são os atributos. 

<input id="email" type="email" required />
        ↑ atributo   ↑ atributo      ↑ atributo sem valor (booleano)

2.4 — A estrutura básica de todo documento

🖥️ Conceito

Este é o mapa mental das 4 camadas de todo documento HTML. Não precisa digitar ainda — só entender: 

<!doctype html>   ← "Navegador, isto é HTML moderno (HTML5)"
<html>            ← a raiz: tudo mora aqui dentro
  <head>          ← os bastidores: infos PARA O NAVEGADOR
  </head>
  <body>          ← o palco: tudo que O USUÁRIO VÊ
  </body>
</html>

Guarde essa dupla: head é o bastidor, body é o palco. O que está no head o público não vê — mas sem bastidor, o show não acontece.

✋ Checkpoint

Antes de seguir, responda para si mesmo: qual é um exemplo de coisa que fica no head? E no body? Se travou aqui, releia a seção 2.4 — esse conceito sustenta todo o resto do guia. Teoria revisada? Então vem a próxima pergunta: onde você ESCREVE esse HTML? É o que vamos instalar agora.

03

Instalando o VSCode do zero

⏱️ ~20 min

O que você vai fazer neste bloco: instalar o VSCode, abrir pela primeira vez e reconhecer as partes principais da interface. Enquanto o download roda, aproveite para ler a próxima seção — o tempo de download é seu aliado.

3.1 — O que é um editor de código (e por que não usar o Word)

Antes de baixar qualquer coisa: por que você precisa de um programa especial para escrever código? Não dá para usar o Word? Não dá — e o motivo é interessante. O Word esconde marcação invisível dentro do arquivo: formatação, fontes, metadados. O navegador leria essa sujeira toda e quebraria.

Código precisa de texto puro. O Bloco de Notas até serviria... mas seria como cozinhar um banquete com uma faca sem ponta. O editor de código é a cozinha profissional: ele colore o código, aponta erros, completa o que você começa a digitar. E o VSCode é o mais usado do mundo — de estagiário a engenheiro do Google. Gratuito, da Microsoft.

3.2 — Download e instalação passo a passo

Atenção: baixe sempre do site oficial — pesquisar "baixar vscode" no Google pode levar a sites falsos com propaganda ou vírus.
⌨️ Faça você

1. Abra o navegador e digite na barra de endereço: code.visualstudio.com

2. Clique no botão grande de Download — o site detecta seu sistema sozinho (Windows, Mac ou Linux).

3. Windows: execute o instalador baixado → aceite o contrato → Next nas telas seguintes, com UMA atenção especial: na tela "Selecionar Tarefas Adicionais", marque as caixas "Adicionar ação 'Abrir com o Code' ao menu de contexto de arquivo" e "...de diretório". Isso cria o atalho de botão direito → Abrir com VSCode, que você vai usar a vida inteira. A opção "Adicionar ao PATH" já vem marcada — deixe assim.

4. Mac: abra o .zip baixado e arraste o ícone do Visual Studio Code para a pasta Aplicativos (Applications). Só isso.

5. Finish/concluir → o VSCode abre pela primeira vez.

Problemas comuns (e como resolver): download bloqueado pelo antivírus (autorize); aviso de "aplicativo de fonte desconhecida" no Mac (botão direito → Abrir); computador sem permissão de administrador (no Windows, o instalador "User Installer" padrão NÃO exige admin — se travar, é outro motivo). Não conseguiu resolver? Anote a dúvida e leve para a aula/grupo.

3.3 — Primeiro contato: tour pela interface

Respira. Essa tela escura cheia de coisa assusta no primeiro dia — em duas semanas ela vai ser sua segunda casa. Por enquanto, você só precisa conhecer 4 partes.

🖥️ Conceito

Encontre cada uma delas na sua tela, uma por vez:

1. Barra lateral esquerda (ícones) — o "menu de ferramentas". O primeiro ícone (folhas) é o Explorer: onde os arquivos do projeto aparecem.

2. Área central — o editor, onde o código é escrito. Hoje mostra a tela de boas-vindas.

3. Ícone de blocos na barra lateral — as Extensões: a "lojinha de aplicativos" do VSCode. Você vai usar já já.

4. Terminal integrado — aperte Ctrl + ' (Ctrl + aspas/crase). Essa janelinha preta que abriu embaixo é um terminal: um jeito de conversar com o computador por texto. Você vai precisar dele daqui a pouco para verificar o Git. Pode fechar no X por enquanto.

3.4 — (Opcional) VSCode em português

O VSCode vem em inglês. Dá para deixar em português instalando uma extensão — MAS um conselho de amigo: tente se acostumar com o inglês. Quase todo tutorial, documentação e mensagem de erro do mundo dev está em inglês, e os menus do editor são um ótimo treino gratuito. Se preferir português, sem problema nenhum: ícone de Extensões → buscar Portuguese (Brazil) Language Pack → Install → reiniciar quando pedir.

✋ Checkpoint

Antes de seguir para o Git: o VSCode está aberto na tela de boas-vindas? Não avance sem isso. Se está tudo certo, aproveite para explorar a tela de boas-vindas por um minuto.

04

Instalando o Git

⏱️ ~20 min

4.1 — O que é Git, em 60 segundos

Você já jogou videogame e salvou o jogo antes de uma fase difícil? Então: o Git é o botão de salvar o jogo do programador. Ele guarda "fotografias" do seu projeto ao longo do tempo. Quebrou tudo? Volta para a fotografia de ontem. Quer saber o que mudou semana passada? Está tudo registrado.

Hoje você só instala e configura. Usar de verdade — os famosos commits — é assunto de uma aula inteira. Mas instalar hoje significa que, na hora certa, está tudo pronto.

4.2 — Download e instalação

⌨️ Faça você

1. Navegue até git-scm.com → botão Download for Windows (ou Mac/Linux).

2. Windows — aviso importante antes de abrir: esse instalador tem umas DEZ telas de opções e você não precisa entender nenhuma agora. A regra é: Next em tudo, com duas exceções:

   • Exceção 1 — tela "Choosing the default editor used by Git": troque o Vim (padrão) por "Use Visual Studio Code as Git's default editor". Por quê? Vim é um editor de terminal que já fez muito marinheiro de primeira viagem ficar preso sem saber sair. Escolha o VSCode que você acabou de instalar.

   • Exceção 2 — tela "Adjusting the name of the initial branch": marque "Override the default branch name" e confirme que está escrito main. Esse é o nome da "linha do tempo" principal do projeto — main é o padrão moderno, o mesmo que o GitHub usa.

   • Todo o resto: Next, Next, Next → Install → Finish.

3. Mac: abra o Terminal (Cmd + Espaço, digite "terminal") e rode git --version. Se o Git não existir, o próprio Mac oferece instalar as ferramentas de desenvolvedor — clique em Instalar e aguarde.

4. Linux: no terminal, sudo apt install git (Ubuntu/Mint e derivados).

4.3 — Verificando se funcionou (seu primeiro comando de terminal!)

⌨️ Faça você

De volta ao VSCode: Ctrl + ' para abrir o terminal integrado. Digite e aperte Enter: 

git --version

Se apareceu algo como git version 2.45..., parabéns: você acabou de executar o primeiro comando de terminal da sua vida. Número de versão diferente do exemplo? Normal. Deu "git não é reconhecido"? Geralmente é só fechar e reabrir o VSCode, porque ele precisa "recarregar" para enxergar programas recém-instalados.

4.4 — Apresentando-se ao Git (configuração de identidade)

Toda fotografia que o Git tira leva uma assinatura: quem fez essa mudança. Então a primeira coisa depois de instalar é se apresentar. São dois comandos — digite com os SEUS dados, não com os do exemplo.

⌨️ Faça você

No terminal, um comando por vez, substituindo pelos seus dados reais — e anote o e-mail usado, porque a conta do GitHub daqui a pouco DEVE usar o mesmo: 

git config --global user.name "Maria Silva"
git config --global user.email "maria.silva@email.com"

Anatomia do comando (sua primeira leitura de "computês"): git chama o programa → config é a ação (configurar) → --global significa "para todos os projetos deste computador" → user.name é qual configuração → e as aspas guardam o valor. Repare: até comando de terminal tem a mesma lógica de nome="valor" dos atributos HTML que você revisou. Programação é padrão se repetindo.

⌨️ Faça você

Conferência final — este comando mostra o que foi salvo:

git config --global --list
✋ Checkpoint

Seu nome e e-mail apareceram na tela do terminal? Erro comum: digitar o comando com aspas "curvas" (copiadas de PDF/WhatsApp) — as aspas precisam ser as retas do teclado.

05

Criando a conta no GitHub

⏱️ ~15 min

5.1 — Git vs GitHub: não é a mesma coisa

Pergunta clássica de iniciante — e de entrevista de estágio: qual a diferença entre Git e GitHub? Analogia: o Git é a câmera fotográfica; o GitHub é o Instagram. O Git tira as "fotos" do projeto aí na sua máquina. O GitHub é o lugar na internet onde você publica essas fotos: para guardar na nuvem, para trabalhar em equipe e — muito importante — para mostrar seu trabalho para o mundo.

O seu perfil do GitHub vai virar o seu portfólio. Recrutador de tecnologia olha GitHub como olha currículo. A conta que você cria hoje é a mesma que, daqui a alguns meses, vai estar no seu LinkedIn. Então capricha.

5.2 — Criando a conta, passo a passo

Segurança: nunca compartilhe sua senha com ninguém, nem digite senhas com a tela sendo projetada ou compartilhada.
⌨️ Faça você

1. Navegue até github.com → botão Sign up (canto superior direito).

2. E-mail: use o mesmo e-mail do git config de 5 minutos atrás. É isso que conecta as fotografias do Git à sua conta.

3. Senha: forte e guardada num lugar seguro (gerenciador de senhas, se você usar).

4. Username: momento importante — esse nome vira seu endereço público: github.com/seunome. Vai para o currículo. Dica: nome profissional, curto, sem números aleatórios. "mariasilva-dev" envelhece melhor que "mariazinha_xx99". Se o nome desejado estiver ocupado, variações com hífen ou sobrenome resolvem.

5. Resolva a verificação "prove que você é humano" (o quebra-cabeça).

6. O GitHub envia um código de confirmação por e-mail — abra seu e-mail em outra aba, copie o código e cole. Não chegou? Confira a caixa de spam.

7. Nas perguntas de personalização ("quantos membros no time?", etc.), pode pular (Skip) ou responder qualquer coisa — não muda nada.

8. No plano, escolha Free. O plano gratuito do GitHub é absurdamente completo — você vai passar o curso inteiro sem pagar nada.

5.3 — Fechando o ciclo do ambiente

Olha o que você tem agora: um editor profissional, uma máquina do tempo instalada e configurada com a sua identidade, e um portfólio público na internet com zero repositórios... por enquanto. Quando você conectar o VSCode ao GitHub pela primeira vez — na aula de Git — o próprio VSCode vai abrir o navegador e pedir para você autorizar com essa conta. Por isso ela precisava existir hoje.

Ambiente pronto. Agora vem a parte boa: usar ele de verdade.

✋ Checkpoint

Pausa de 10 minutos — você merece. Este é o corte natural do guia: fim da instalação, começo do código. Se ficou alguma pendência (download travado, e-mail que não chegou), este é o momento de resolver com calma. E se o GitHub não funcionou ainda, tudo bem: você não precisa dele para acompanhar o resto do guia — dá para resolver depois.

06

Mão na massa: criando a pasta e o projeto

⏱️ ~15 min

O que você vai fazer neste bloco: criar a pasta do projeto, abri-la no VSCode e criar o arquivo index.html. Não avance sem esses três itens prontos.

6.1 — Criando a pasta do projeto

Regra de ouro que você vai levar para a vida: todo projeto vive dentro de uma pasta própria. Nunca, jamais, crie arquivo solto na Área de Trabalho ou dentro de Downloads. Projeto é como gaveta: cada um tem a sua.

⌨️ Faça você

1. Abra o Explorador de Arquivos (Windows) ou Finder (Mac).

2. Navegue até Documentos e crie uma pasta chamada projetos (se ainda não existir).

3. Dentro dela, crie a pasta bem-cuidado.

Nomenclatura desde já (previne dor de cabeça futura): nomes de pastas e arquivos sempre em minúsculas, sem espaços (use hífen: bem-cuidado, não Bem Cuidado), sem acentos ou ç. Espaços e acentos quebram links e comandos no futuro — melhor criar o hábito hoje.

6.2 — Abrindo a pasta no VSCode + o conceito de pasta raiz

⌨️ Faça você

No VSCode: File → Open Folder (Arquivo → Abrir Pasta) e selecione a pasta bem-cuidado. Atalho: Ctrl + K, Ctrl + O.

Perceba uma coisa importante: você abriu a pasta, não um arquivo. O VSCode agora enxerga essa pasta como o universo do seu projeto. Essa pasta principal tem um nome: pasta raiz — em inglês, root.

Por que "raiz"? Pense numa árvore de cabeça para baixo: a raiz fica no topo, e tudo cresce a partir dela. Toda subpasta — imagens, estilos, scripts — vai nascer de dentro da raiz. Quando alguém falar "na raiz do projeto", é aqui: o primeiro nível da pasta principal.

6.3 — Criando o index.html

⌨️ Faça você

No painel esquerdo (Explorer), passe o mouse sobre o nome da pasta → clique no ícone New File (folhinha com +) → digite index.html e Enter.

Por que index? Convenção histórica e universal: quando alguém acessa um site tipo meusite.com, o servidor procura automaticamente um arquivo chamado index.html para mostrar. Index é o "índice", a porta de entrada. Toda página inicial de projeto se chama index — grave isso.

E a extensão .html diz ao computador e ao VSCode que idioma esse arquivo fala. Repare que o ícone do arquivo já mudou — o VSCode entendeu.

✋ Checkpoint

Antes de codar: o index.html está aberto e o nome da pasta bem-cuidado aparece no topo do Explorer? Se algo travou, resolva agora — a partir daqui é digitação contínua.

07

O esqueleto do documento (e a mágica do Emmet)

⏱️ ~20 min

7.1 — Primeiro à mão, depois com atalho

Importante: na primeira vez, digite o esqueleto à mão, linha por linha. Só depois você vai conhecer o atalho. Quem aprende o atalho antes do conceito nunca entende o que o atalho gera — não pule esta etapa!

⌨️ Faça você

Digite linha por linha, lendo os comentários de cada uma. Sem pressa — velocidade vem com o tempo: 

<!-- Declara que este documento é HTML5 -->
<!doctype html>

<!-- Raiz do documento. lang="pt-BR" avisa o idioma da página -->
<html lang="pt-BR">
  <head>
    <!-- charset UTF-8: garante acentos e ç funcionando -->
    <meta charset="UTF-8" />
    <!-- viewport: faz a página se adaptar ao celular -->
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <!-- Texto que aparece na aba do navegador -->
    <title>Minhas Consultas · Organizador Médico</title>
  </head>
  <body>
    <!-- Tudo que o usuário vê entra aqui -->
  </body>
</html>

7.2 — Entendendo linha por linha

Linha O que ela significa
<!doctype html> É o crachá do documento. A primeira coisa que o navegador lê. Sem ele, o navegador entra num "modo antigo" e pode renderizar tudo torto. Não é uma tag de verdade — é uma declaração, por isso não fecha.
lang="pt-BR" Seu primeiro atributo em ação! Ele avisa o idioma para três públicos: leitores de tela (acessibilidade — a voz lê com o sotaque certo), o Google (SEO) e o corretor do navegador.
<meta charset="UTF-8" /> Meta = informação sobre a página. UTF-8 é a tabela de caracteres que inclui ç, ã, é... Esquecer essa linha = página cheia de símbolos estranhos tipo "GestÃ£o". Você provavelmente já viu isso em algum site.
<meta name="viewport" ... /> Traduzindo o content: "a largura da página = largura do aparelho, começando com zoom de 100%". É o que impede o site de aparecer minúsculo no celular.
<title> O texto da abinha do navegador. Também é o que aparece de azul nos resultados do Google. Único texto "visível" que mora no head.

7.3 — Agora sim: a mágica do Emmet

Você digitou tudo à mão, sofreu um pouquinho... agora vem um segredo. Apague TUDO. Confia.

⌨️ Faça você

Com o arquivo vazio, digite apenas ! e aperte Tab (ou Enter na sugestão).

💥 O esqueleto inteiro aparece. Pode se impressionar.

Isso se chama Emmet — vem instalado no VSCode. Ele expande abreviações em HTML completo. Mas repare num detalhe: ele gerou lang="en". O Emmet não sabe que você é brasileiro — troque para pt-BR. Atalho nunca substitui atenção.

7.4 — Ver no navegador pela primeira vez

⌨️ Faça você

Dentro do <body>, escreva: <h1>Olá, mundo!</h1>. Salve com Ctrl + S.

Depois: botão direito no arquivo → Reveal in File Explorer → dois cliques no index.html para abrir no navegador.

Parabéns: você acabou de publicar sua primeira página... para o público de uma pessoa. 😄 Mais adiante no guia tem um jeito bem melhor de visualizar isso, com atualização automática. Por enquanto, lembre: mudou o código → Ctrl+S → F5 no navegador.

✋ Checkpoint

Pausa natural. O "Olá, mundo!" apareceu no navegador? Se não: os erros mais comuns são esquecer de salvar, criar o arquivo fora da pasta ou digitar index.htm (sem o L).

08

Construindo o Bem-cuidado, seção por seção

⏱️ ~60 min (coração do guia)

Método: para cada parte — (1) leia o conceito, (2) digite o código, (3) salve e veja no navegador. O feedback visual constante é o que mantém a motivação. Nunca digite mais de ~10 linhas sem olhar o resultado.

8.1 — Head completo: conectando fontes externas

🖥️ Conceito

Antes de digitar, entenda o conceito de recursos externos: o HTML pode "chamar" arquivos de fora — folhas de estilo, fontes, imagens. É o que você vai fazer agora, dentro do head: 

<!-- Link de conexão com o arquivo de estilos (você criará na aula de CSS) -->
<link rel="stylesheet" href="styles.css" />

<!-- Importação de fonte do Google Fonts -->
<link rel="preconnect" href="https://fonts.googleapis.com" />
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin />
<link href="https://fonts.googleapis.com/css2?family=Inter:wght@400;600;700&display=swap" rel="stylesheet" />

A tag <link> é o "cabo" que conecta seu HTML a outros arquivos. Dois atributos importam: rel diz a relação ("esse arquivo é uma folha de estilo") e href diz o endereço.

E esse preconnect? Analogia: é como ligar para o restaurante avisando que você está chegando — quando você chega, a mesa já está pronta. O navegador abre a conexão com o servidor de fontes ANTES de precisar dela. Página mais rápida.

O styles.css ainda nem existe — e está tudo bem! O navegador procura, não acha, e segue a vida. É o gancho para a aula de CSS.

8.2 — Header: o topo da página + navegação

⌨️ Faça você

Dentro do body:

<!-- <header> = identidade do site / saudação -->
<header class="topo">
  <div class="logo">🩺 Bem-cuidado</div>
  <nav aria-label="principal">
    <!-- Links âncora (#) levam para seções da MESMA página -->
    <a href="#proximas">Próximas</a>
    <a href="#historico">Histórico</a>
    <a href="#anexos">Exames</a>
  </nav>
</header>

Entendendo linha a linha:

  • <header> — é uma tag semântica: o nome dela já diz o que ela é. Header = cabeçalho. Você poderia usar uma <div>? Poderia. Mas div não diz nada — é uma caixa sem etiqueta. Tags semânticas ajudam o Google e leitores de tela a entenderem a página.
  • class="topo" — class é um atributo de "apelido". Não muda nada visualmente hoje, mas é o gancho que o CSS vai usar para estilizar. Você está deixando pontos de encaixe prontos.
  • <nav> — outra semântica: navegação. E o aria-label="principal" é acessibilidade: um leitor de tela anuncia "navegação principal" para quem não enxerga a tela.
  • <a href="#proximas"> — o "a" vem de anchor, âncora. Quando o href começa com #, o link não vai para outro site: ele rola a página até o elemento que tiver id="proximas". Você ainda vai criar esse id — o link é a fechadura, o id é a chave.
✋ Checkpoint

Pense: se eu clicar em "Histórico" agora, o que acontece? Resposta: nada visível ainda, porque não existe elemento com id="historico". Guarde essa relação href ↔ id — ela vai aparecer já já.

8.3 — Main + a primeira section

⌨️ Faça você

Abaixo do header, ainda dentro do body:

<!-- <main> deve existir UMA única vez por página -->
<main>
  <section class="saudacao">
    <h1>Olá, Dona Lúcia 👋</h1>
    <p>Você tem <strong>2 consultas</strong> nesta semana.</p>
  </section>
</main>

Três regras de ouro aqui. Um: <main> é o conteúdo principal e só pode existir um por página — como só existe um miolo num livro. Dois: <h1> é o título mais importante e a boa prática é ter apenas um h1 por página. Os títulos vão de h1 a h6, como capítulos e subcapítulos. Três: <strong> não é só "negrito" — é ênfase de importância. O visual é consequência; o significado vem primeiro.

8.4 — Próximas consultas: section, article e time

⌨️ Faça você

Ainda dentro do main, abaixo da saudação:

<section id="proximas" aria-labelledby="t-proximas">
  <h2 id="t-proximas">Próximas consultas</h2>

  <!-- Cada consulta é um <article>: faz sentido por si só -->
  <article class="consulta">
    <time datetime="2026-07-08T14:30">Qua, 08/jul · 14h30</time>
    <h3>Dra. Marina Couto — Cardiologia</h3>
    <p>Hospital São Lucas · Sala 312</p>
    <p class="lembrete">Levar exames de sangue recentes.</p>
  </article>
</section>

Conceitos-chave (este é o trecho mais rico do guia — leia com atenção):

  • section vs article vs div — section é um capítulo temático da página. Article é um conteúdo que faz sentido sozinho, fora do contexto: se você recortar essa consulta e colar num e-mail, ela ainda se explica — tem data, médico, local. Isso é um article. Já a div é a caixa genérica: use quando nenhuma tag semântica servir.
  • O id apareceu! — Lembra do link #proximas no menu? Você acabou de criar a chave. Salve, atualize o navegador e clique no link do menu. Funcionou? Isso é a dupla href/id em ação. E atenção: id é único — não podem existir dois elementos com o mesmo id na página. É como CPF.
  • aria-labelledby — conecta a section ao seu título pelo id. Leitor de tela anuncia: "seção: Próximas consultas". Acessibilidade não é extra — no nosso projeto, a Dona Lúcia pode ter baixa visão. A gente constrói para todo mundo desde a primeira linha.
  • <time> com datetime — para o humano, mostramos "Qua, 08/jul · 14h30" — bonitinho, em português. Para a máquina, o atributo datetime carrega o formato ISO: 2026-07-08T14:30, ano-mês-dia T hora. Por que os dois? Porque buscadores e leitores de tela entendem o formato padrão, e o usuário lê o formato amigável. É a mesma informação em dois idiomas: humanês e computês.
⌨️ Faça você

Desafio rápido (sozinho, ~3 min): crie VOCÊ o segundo article: um Ultrassom abdominal, sexta 10/jul às 09h00, na Clínica Imagem, com a observação "Jejum de 8h". Use o primeiro como modelo — e só depois compare com o que você fez.

8.5 — Histórico: quando usar tabela

Tabela em HTML tem má fama porque antigamente usavam para montar o layout inteiro do site — era uma gambiarra. A regra moderna é simples: tabela serve para dados tabulares — informação que naturalmente vive em linhas e colunas, tipo planilha. Histórico de consultas com data, profissional e observação? Caso perfeito.

⌨️ Faça você

Abaixo da section de próximas consultas:

<section id="historico" aria-labelledby="t-historico">
  <h2 id="t-historico">Histórico médico</h2>

  <table>
    <caption>Últimas consultas realizadas</caption>
    <thead>
      <tr>
        <th scope="col">Data</th>
        <th scope="col">Profissional</th>
        <th scope="col">Observações</th>
      </tr>
    </thead>
    <tbody>
      <tr>
        <td>12/05/2026</td>
        <td>Dr. André Lima — Clínico Geral</td>
        <td>Pressão controlada. Manter medicação.</td>
      </tr>
    </tbody>
  </table>
</section>

Anatomia da tabela — a analogia da planilha:

Tag Tradução Como pensar
<table> a planilha A moldura de tudo
<caption> o título da planilha Descreve a tabela — leitores de tela agradecem
<thead> / <tbody> cabeçalho / corpo Separa a linha de títulos das linhas de dados
<tr> table row = linha Toda célula mora dentro de uma linha
<th> table header = célula-título O scope="col" diz: "sou título da coluna"
<td> table data = célula de dado O conteúdo em si
⌨️ Faça você

Desafio rápido: adicione a segunda linha do tbody sozinho — 02/04/2026 · Dra. Helena Souza — Oftalmo · Trocar grau dos óculos.

8.6 — Exames anexados: lista + botão

⌨️ Faça você

Abaixo do histórico:

<section id="anexos" aria-labelledby="t-anexos">
  <h2 id="t-anexos">Exames anexados</h2>
  <ul class="anexos">
    <li>📄 Hemograma — 02/04/2026</li>
    <li>📄 Eletrocardiograma — 14/03/2026</li>
    <li>📄 Raio-X de tórax — 28/01/2026</li>
  </ul>
  <!-- Em um projeto futuro este botão abrirá um seletor de arquivo -->
  <button type="button">+ Anexar novo exame</button>
</section>

Duas listas existem no HTML: <ul>, unordered list, com bolinhas — para quando a ordem não importa, tipo lista de compras. E <ol>, ordered list, numerada — para quando a ordem importa, tipo passo a passo de receita. Cada item, nas duas, é um <li>, list item.

E o botão? Repare no type="button". Botão sem type, dentro de formulário, tenta ENVIAR o formulário por padrão. Esse aqui é só visual por enquanto — na aula de JavaScript ele vai ganhar vida e abrir um seletor de arquivos. HTML é a estrutura; o comportamento vem depois.

8.7 — Formulário: a parte mais importante para o futuro

🖥️ Conceito

Formulário é como o site OUVE o usuário. Login, cadastro, busca, compra — tudo é form. Atenção dobrada nesta seção.

⌨️ Faça você

Abaixo da section de anexos:

<section aria-labelledby="t-prefs">
  <h2 id="t-prefs">Receber lembretes por e-mail</h2>

  <form action="mailto:lembretes@bemcuidado.app" method="post" enctype="text/plain">
    <p>
      <label for="email">Seu e-mail</label><br />
      <input id="email" name="email" type="email" required
             placeholder="voce@exemplo.com" />
    </p>
    <p>
      <label for="quando">Quando avisar</label><br />
      <select id="quando" name="quando">
        <option>24 horas antes</option>
        <option>2 horas antes</option>
        <option>No dia anterior pela manhã</option>
      </select>
    </p>
    <button type="submit">Quero receber lembretes</button>
  </form>
</section>

Pontos essenciais:

  • action + method — action é o destino: para onde os dados vão quando o formulário for enviado. Aqui usamos mailto:, que abre o app de e-mail do computador — um truque didático, porque ainda não temos servidor. No mundo real, o action aponta para um endereço do back-end... que VOCÊ vai construir lá na frente do curso. Method é o "como" — GET e POST serão detalhados na aula de formulários.
  • A dupla label + input — olha o padrão: <label for="email"> aponta para o <input id="email">. Mesmo valor nos dois. Resultado: clicar no TEXTO do rótulo já foca o campo (teste!), e o leitor de tela anuncia o rótulo certo. label for + input id = acessibilidade total. Nunca crie input sem label.
  • type="email" — o input muda de personalidade conforme o type: email, password, date, number... Com type email, o navegador VALIDA sozinho: tente enviar "banana" no campo e veja o navegador reclamar. Validação de graça, sem uma linha de JavaScript.
  • required — atributo booleano: a presença dele já significa "sim, obrigatório". Não precisa de valor.
  • name — o id é para a página (label, CSS, âncoras). O name é a etiqueta que viaja junto com o dado quando o formulário é enviado — é como o servidor sabe qual campo é qual.
  • select + option — caixa de seleção: o select é a caixa, cada option é uma escolha. Quando as opções são fixas e poucas, é melhor que campo livre — ninguém digita errado.
⌨️ Faça você

Teste ao vivo: salve, atualize o navegador e (1) clique no label para ver o foco pular para o campo, (2) tente enviar com o campo vazio, (3) tente enviar "banana". Descubra as mensagens de erro do navegador — elas trabalham de graça para você.

8.8 — Fechando com o footer

⌨️ Faça você

Depois do fechamento do main (fora dele!):

<footer>
  <p>Feito com 💙 pela equipe Bem-cuidado · 2026</p>
</footer>

Footer, o rodapé — fecha o trio semântico com header e main. Repare que ele fica FORA do main: ele não é conteúdo principal, é o fechamento da página. E... pronto. A casa da Dona Lúcia está de pé! Sem pintura, sem decoração — isso é papel do CSS — mas com uma estrutura sólida, semântica e acessível.

09

Analogias para fixar

⏱️ revisão

Use esta tabela como material de revisão: sempre que um conceito embolar, volte aqui e releia a analogia. Melhor ainda: tente explicar cada uma para outra pessoa — ensinar é o jeito mais rápido de aprender.

Conceito Analogia Frase para lembrar
HTML / CSS / JS Casa "HTML é a estrutura, CSS é a decoração, JS é a eletricidade."
Documento HTML Receita de bolo "O head é a lista de ingredientes e o forno pré-aquecido (preparação que ninguém come); o body é o modo de preparo, o passo a passo que vira bolo. E o navegador é o cozinheiro: ele lê de cima para baixo, na ordem, e executa. Se a instrução está errada, o bolo sai errado — o cozinheiro não adivinha."
Hierarquia h1–h6 e seções Livro "h1 é o título do livro (só existe um na capa), h2 são os capítulos, h3 os subcapítulos. Ninguém escreve o capítulo 3 antes do título do livro — a hierarquia comunica importância."
Tags aninhadas Caixas dentro de caixas / matryoshka "A caixa que abriu por último fecha primeiro. Se abri section e depois article, fecho article e depois section. Caixas não se atravessam."
Atributos Etiqueta de roupa "A peça é a tag (camiseta); tamanho, cor e instruções de lavagem são os atributos."
href ↔ id Chave e fechadura "O link com # é a fechadura procurando; o id é a chave que encaixa. Sem os dois, a porta não abre."
id vs class CPF vs sobrenome "id é CPF: único, identifica UM elemento. class é sobrenome: várias pessoas podem compartilhar. Vários articles podem ter class='consulta'; só um pode ter id='proximas'."
Semântica (header, nav, main...) Potes de cozinha rotulados "Div é um pote sem rótulo: funciona, mas você precisa abrir para saber o que tem. Tags semânticas são potes rotulados: qualquer um (Google, leitor de tela, o colega que pegar seu código) acha o açúcar de primeira."
Indentação / aninhamento Lista de tópicos e subtópicos "Quanto mais para dentro o recuo, mais 'filho' o elemento é. É a árvore genealógica do documento — e aqui já começa a lógica de programação: hierarquia, dependência, escopo."
Ponte para a lógica: repare que o tempo todo você pensou em hierarquia (o que está dentro do quê), sequência (o navegador lê de cima para baixo) e regras rígidas (abriu, tem que fechar). Isso É pensamento de programação. Quando chegar em JavaScript, você vai manipular exatamente essa árvore que construiu hoje — ela tem até nome: DOM.

10.2 — Extensão Prettier (o arrumador automático)

Prettier é o funcionário que arruma sua mesa toda vez que você sai da sala. Ele formata o código automaticamente: indentação, espaçamento, tudo padronizado.

  • Instalar: busque Prettier – Code formatter nas extensões.
  • Ativar formatação ao salvar: Ctrl + , (Configurações) → busque format on save → marque. Depois busque default formatter → escolha Prettier.
  • Teste você mesmo: desalinhe algumas linhas de propósito, salve, e veja tudo se arrumar sozinho.

10.3 — Emmet além do "!"

🖥️ Conceito

Teste cada abreviação + Tab num arquivo de rascunho. Comece pelas duas primeiras; as outras são "sementes" para você descobrir depois:

Você digita Emmet gera
! Esqueleto HTML5 completo
h1 <h1></h1> — qualquer tag pelo nome
ul>li*3 Lista com 3 itens (o > = "dentro de", o * = "vezes")
section#anexos <section id="anexos"> — # vira id
p.lembrete <p class="lembrete"> — ponto vira class
table>tr*2>td*3 Tabela 2 linhas × 3 células

Atalho O que faz

11

Boas práticas desde o primeiro dia

⏱️ ~10 min

Última parte teórica, e talvez a mais importante. Código funciona ou não funciona — mas código BOM é outra coisa: é código que outra pessoa entende. E essa "outra pessoa" inclui você mesmo daqui a três meses, olhando para o próprio código sem lembrar de nada.

As 6 regras de ouro (anote e cole perto do computador)

  1. Indentação sempre. Elemento filho entra um "degrau" (Tab) para dentro do pai. O navegador ignora, mas o humano não: indentação é o mapa visual do aninhamento. E com Prettier + Shift Alt F, não há desculpa.
  2. Nomes de arquivo: minúsculos, sem espaço, sem acento. index.html, sobre-mim.html, styles.css. Nunca Página Inicial.html.
  3. Semântica antes de div. Antes de digitar <div>, pergunte: existe uma tag que descreve isso? Header, nav, main, section, article, footer, time...
  4. Toda tag que abre, fecha — e na ordem certa (a última que abriu é a primeira que fecha).
  5. Comente o porquê, não o óbvio. Olhe os comentários do próprio projeto: <!-- Links âncora (#) levam para seções da MESMA página --> explica uma decisão. Já <!-- parágrafo --> em cima de um <p> é ruído.
  6. Acessibilidade não é opcional. label com for, aria-label na navegação, lang no html, alt em imagens (spoiler da próxima aula). Construímos para a Dona Lúcia — e para todo mundo.
Sobre erros: se o seu código quebrou hoje, parabéns: você programou. Dev sênior não é quem não erra — é quem acha o erro rápido. Esqueceu de fechar uma tag e a página inteira ficou estranha? Todo dev do mundo já passou ou vai passar por isso. O erro é o seu professor particular.
