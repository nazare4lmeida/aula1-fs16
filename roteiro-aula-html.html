01

Abertura + relembrando: o que é HTML?

⏱️ ~10 min

Objetivo do bloco: reativar a memória da aula passada em 10 minutos, sem reexplicar tudo — perguntas rápidas valem mais que monólogo. A revisão vem ANTES da instalação de propósito: relembra o "porquê" antes do "como", e a turma chega na parte técnica sabendo para que serve cada ferramenta.

"Bom, pessoal! Aula passada foi teoria: o que é HTML, tags, atributos... Hoje é o grande dia de sujar as mãos — e do jeito completo: a gente vai montar o ambiente de desenvolvimento de vocês DO ZERO — editor, Git, conta no GitHub — e, com tudo pronto, escrever a primeira página de verdade. No fim da aula, vocês saem daqui com o computador de um dev e um site funcionando nele."

"Mas antes, teste rápido de memória — sem medo de errar, tá? Quem me diz: o que significa a sigla HTML?"

✋ Pausa

Deixe a turma tentar. Resposta esperada: HyperText Markup Language — Linguagem de Marcação de Hipertexto. Se ninguém lembrar, dê a primeira palavra e deixe completarem.

Recapitulação em 3 frases (a analogia da casa)

"Lembram da nossa casa? Um site é como uma casa: o HTML é a estrutura — paredes, portas, cômodos. O CSS, que vem nas próximas aulas, é a decoração: cor da parede, móveis, estilo. E o JavaScript é a parte elétrica e o encanamento: o que faz a casa funcionar — luz acende, campainha toca."

"Então hoje a gente constrói só a estrutura. E olha: uma casa sem pintura é feia, mas fica de pé. Uma casa sem estrutura... nem existe. Por isso HTML vem primeiro."

Ponto para reforçar: HTML não é linguagem de programação — é linguagem de marcação. Ela não toma decisões, não faz contas. Ela descreve e organiza conteúdo: "isto é um título", "isto é um parágrafo", "isto é uma tabela".

"Pensa num texto no Word onde você seleciona algo e marca como 'Título 1'. O HTML faz isso, só que escrevendo a marcação à mão. A gente não está programando lógica ainda — a gente está etiquetando conteúdo pro navegador entender."

O que vamos construir hoje

🖥️ Mostrar

Descreva (sem mostrar o código!) o projeto: "Bem-cuidado", um organizador de consultas médicas para a Dona Lúcia — com próximas consultas, histórico, exames anexados e um formulário de lembretes.

Dica: pergunte "quem aqui tem avó/mãe que vive perdendo papelzinho de consulta?" — conectar o projeto com a vida real aumenta o engajamento.

02

Revisão-relâmpago: tags, elementos e atributos

⏱️ ~15 min

Objetivo do bloco: revisar os 4 conceitos-chave da aula passada em formato de "quiz relâmpago". Você pergunta, eles respondem, você só corrige e completa.

2.1 — Tag: a etiqueta

"Tag, em inglês, é etiqueta. É literalmente isso: uma etiqueta que você cola no conteúdo pra dizer o que ele é. Toda tag vem entre os sinais de menor e maior: <p>, <h1>, <table>."

"E quase toda tag anda em dupla: a que abre e a que fecha. A que fecha tem uma barra: </p>. É como abrir e fechar aspas num texto — se você abre e não fecha, vira bagunça."

🖥️ Mostrar

Escreva no editor (pode ser num arquivo de rascunho) e desmonte a anatomia na frente deles:

<p>Você tem 2 consultas nesta semana.</p>
↑ abre     ↑ conteúdo                    ↑ fecha

2.2 — Elemento: o pacote completo

Elemento = tag de abertura + conteúdo + tag de fechamento. A tag é a etiqueta; o elemento é a caixa inteira com a etiqueta colada.

"Uma exceção pra vocês guardarem: algumas tags não têm conteúdo dentro, então não precisam fechar. São as tags vazias, como <br /> (quebra de linha), <input /> (campo de formulário) e <meta />. Elas são autossuficientes — pensa numa etiqueta que já é o produto inteiro."

2.3 — Atributo: a informação extra

"Atributo é uma informação extra que a gente escreve dentro da tag de abertura, sempre no formato nome="valor". Pensa assim: a tag diz o que o elemento é; o atributo diz como ele é ou detalhes sobre ele."

"Exemplo do mundo real: 'camiseta' é a tag. Tamanho M, cor azul — são os atributos."

<input id="email" type="email" required />
        ↑ atributo   ↑ atributo      ↑ atributo sem valor (booleano)

2.4 — A estrutura básica de todo documento

🖥️ Mostrar

Desenhe (pode ser à mão, no quadro ou num diagrama) as 4 camadas. Não digite ainda — é só o mapa mental:

<!doctype html>   ← "Navegador, isto é HTML moderno (HTML5)"
<html>            ← a raiz: tudo mora aqui dentro
  <head>          ← os bastidores: infos PARA O NAVEGADOR
  </head>
  <body>          ← o palco: tudo que O USUÁRIO VÊ
  </body>
</html>

"Guardem essa dupla: head é o bastidor, body é o palco. O que está no head o público não vê — mas sem bastidor, o show não acontece."

✋ Pausa

Checagem rápida antes de seguir: "Alguém me dá um exemplo de coisa que fica no head? E no body?" Se a turma travar aqui, vale mais 2 minutos de revisão — esse conceito sustenta o resto da aula. Gancho para o próximo bloco: "Teoria revisada. Mas onde a gente ESCREVE esse HTML? É o que vamos instalar agora."

03

Instalando o VSCode do zero

⏱️ ~20 min

Objetivo do bloco: todo mundo com o VSCode instalado, aberto e reconhecendo a interface. Enquanto os downloads rodam, aproveite pra conversar — o tempo de download é seu aliado, não inimigo.

3.1 — O que é um editor de código (e por que não usar o Word)

"Antes de baixar qualquer coisa: por que a gente precisa de um programa especial pra escrever código? Não dá pra usar o Word? Não dá — e o motivo é interessante. O Word esconde marcação invisível dentro do arquivo: formatação, fontes, metadados. O navegador leria essa sujeira toda e quebraria."

"Código precisa de texto puro. O Bloco de Notas até serviria... mas seria como cozinhar um banquete com uma faca sem ponta. O editor de código é a cozinha profissional: ele colore o código, aponta erros, completa o que você começa a digitar. E o VSCode é o mais usado do mundo — de estagiário a engenheiro do Google. Gratuito, da Microsoft."

3.2 — Download e instalação passo a passo

🖥️ Mostrar

Projete o site oficial e faça cada clique na tela ANTES de mandar a turma repetir. Alerte: "baixem sempre do site oficial — pesquisar 'baixar vscode' no Google pode levar a sites falsos com propaganda ou vírus."

⌨️ Digitar junto

1. Abram o navegador e digitem na barra de endereço: code.visualstudio.com

2. Cliquem no botão grande de Download — o site detecta o sistema sozinho (Windows, Mac ou Linux).

3. Windows: executem o instalador baixado → aceitem o contrato → Next nas telas seguintes, com UMA atenção especial: na tela "Selecionar Tarefas Adicionais", marquem as caixas "Adicionar ação 'Abrir com o Code' ao menu de contexto de arquivo" e "...de diretório". Expliquem que isso cria o atalho de botão direito → Abrir com VSCode, que vamos usar a vida inteira. A opção "Adicionar ao PATH" já vem marcada — deixem assim.

4. Mac: abram o .zip baixado e arrastem o ícone do Visual Studio Code para a pasta Aplicativos (Applications). Só isso.

5. Finish/concluir → o VSCode abre pela primeira vez.

Problemas comuns (circule pela sala): download bloqueado pelo antivírus (autorizar); aviso de "aplicativo de fonte desconhecida" no Mac (botão direito → Abrir); computador sem permissão de administrador (no Windows, o instalador "User Installer" padrão NÃO exige admin — se travar, é outro motivo).

3.3 — Primeiro contato: tour guiado pela interface

"Respira. Essa tela escura cheia de coisa assusta no primeiro dia — em duas semanas ela vai ser sua segunda casa. Vamos nomear só as 4 partes que importam agora."

🖥️ Mostrar

Aponte na tela, um por vez, e peça pra acharem no computador deles:

1. Barra lateral esquerda (ícones) — o "menu de ferramentas". O primeiro ícone (folhas) é o Explorer: onde os arquivos do projeto aparecem.

2. Área central — o editor, onde o código é escrito. Hoje mostra a tela de boas-vindas.

3. Ícone de blocos na barra lateral — as Extensões: a "lojinha de aplicativos" do VSCode. Vamos usar já já.

4. Terminal integrado — apertem Ctrl + ' (Ctrl + aspas/crase). "Essa janelinha preta que abriu embaixo é um terminal: um jeito de conversar com o computador por texto. Vamos precisar dele daqui a pouco pra verificar o Git. Pode fechar no X por enquanto."

3.4 — (Opcional) VSCode em português

"O VSCode vem em inglês. Dá pra deixar em português instalando uma extensão — vou mostrar como, MAS um conselho de amigo: tentem se acostumar com o inglês. Quase todo tutorial, documentação e mensagem de erro do mundo dev está em inglês, e os menus do editor são um ótimo treino gratuito. Quem preferir português, sem problema nenhum: ícone de Extensões → buscar Portuguese (Brazil) Language Pack → Install → reiniciar quando pedir."

✋ Pausa

Checkpoint obrigatório: "Levanta a mão quem está com o VSCode aberto na tela de boas-vindas." Ninguém segue para o Git sem isso. Alunos prontos podem explorar a tela de boas-vindas enquanto você atende os travados.

04

Instalando o Git

⏱️ ~20 min

4.1 — O que é Git, em 60 segundos

"Quem aqui já jogou videogame e salvou o jogo antes de uma fase difícil? Então: o Git é o botão de salvar o jogo do programador. Ele guarda 'fotografias' do seu projeto ao longo do tempo. Quebrou tudo? Volta pra fotografia de ontem. Quer saber o que mudou semana passada? Está tudo registrado."

"Hoje a gente só instala e configura. Usar de verdade — os famosos commits — é assunto de uma aula inteira. Mas instalar hoje significa que na hora certa está tudo pronto."

4.2 — Download e instalação

⌨️ Digitar junto

1. Naveguem até git-scm.com → botão Download for Windows (ou Mac/Linux).

2. Windows — atenção, aviso importante antes de abrir: "esse instalador tem umas DEZ telas de opções e vocês não precisam entender nenhuma agora. A regra é: Next em tudo, com duas exceções que eu vou avisar."

   • Exceção 1 — tela "Choosing the default editor used by Git": troquem o Vim (padrão) por "Use Visual Studio Code as Git's default editor". Explique: "Vim é um editor de terminal que já fez muito marinheiro de primeira viagem ficar preso sem saber sair. Escolham o VSCode que acabamos de instalar."

   • Exceção 2 — tela "Adjusting the name of the initial branch": marquem "Override the default branch name" e confirmem que está escrito main. Explique: "É o nome da 'linha do tempo' principal do projeto — main é o padrão moderno, o mesmo que o GitHub usa."

   • Todo o resto: Next, Next, Next → Install → Finish.

3. Mac: abram o Terminal (Cmd + Espaço, digitem "terminal") e rodem git --version. Se o Git não existir, o próprio Mac oferece instalar as ferramentas de desenvolvedor — cliquem em Instalar e aguardem.

4. Linux: no terminal, sudo apt install git (Ubuntu/Mint e derivados).

4.3 — Verificando se funcionou (primeiro comando de terminal da turma!)

⌨️ Digitar junto

De volta ao VSCode: Ctrl + ' para abrir o terminal integrado. Todos digitam e apertam Enter:

git --version

"Se apareceu algo como git version 2.45..., parabéns: vocês acabaram de executar o primeiro comando de terminal da vida de vocês. Número diferente do meu? Normal, é só a versão. Deu 'git não é reconhecido'? Me chama — geralmente é só fechar e reabrir o VSCode, porque ele precisa 'recarregar' pra enxergar programas recém-instalados."

4.4 — Apresentando-se ao Git (configuração de identidade)

"Toda fotografia que o Git tira leva uma assinatura: quem fez essa mudança. Então a primeira coisa depois de instalar é se apresentar. São dois comandos, e vocês vão digitar com os SEUS dados, não com os meus."

⌨️ Digitar junto

No terminal, um comando por vez (substituindo pelos dados reais — e anotem o e-mail usado, porque a conta do GitHub daqui a pouco DEVE usar o mesmo):

git config --global user.name "Maria Silva"
git config --global user.email "maria.silva@email.com"

Explicar a anatomia do comando (primeira leitura de "computês" da turma): git chama o programa → config é a ação (configurar) → --global significa "para todos os projetos deste computador" → user.name é qual configuração → e as aspas guardam o valor. "Reparem: até comando de terminal tem a mesma lógica de nome="valor" dos atributos HTML que revisamos. Programação é padrão se repetindo."

⌨️ Digitar junto

Conferência final — este comando mostra o que foi salvo:

git config --global --list
✋ Pausa

Checkpoint: "Todo mundo vendo o próprio nome e e-mail na tela do terminal?" Erro comum: digitar o comando com aspas "curvas" (copiou de PDF/WhatsApp) — as aspas precisam ser as retas do teclado.

05

Criando a conta no GitHub

⏱️ ~15 min

5.1 — Git vs GitHub: não é a mesma coisa

"Pergunta clássica de iniciante — e de entrevista de estágio: qual a diferença entre Git e GitHub? Analogia: o Git é a câmera fotográfica; o GitHub é o Instagram. O Git tira as 'fotos' do projeto aí na sua máquina. O GitHub é o lugar na internet onde você publica essas fotos: pra guardar na nuvem, pra trabalhar em equipe e — muito importante pra vocês — pra mostrar seu trabalho pro mundo."

"O perfil do GitHub de vocês vai virar o portfólio de vocês. Recrutador de tecnologia olha GitHub como olha currículo. A conta que a gente cria hoje é a mesma que, daqui a alguns meses, vai estar no LinkedIn de vocês. Então capricha."

5.2 — Criando a conta, passo a passo

🖥️ Mostrar

Faça o fluxo na tela com uma conta de demonstração descartável. Aviso de privacidade: peça para a turma digitar as próprias senhas SEM você projetar a tela deles, óbvio — e nunca peça pra ninguém falar senha em voz alta.

⌨️ Digitar junto

1. Naveguem até github.com → botão Sign up (canto superior direito).

2. E-mail: usem o mesmo e-mail do git config de 5 minutos atrás. É isso que conecta as fotografias do Git à conta de vocês.

3. Senha: forte e guardada num lugar seguro (gerenciador de senhas, se usarem).

4. Username: momento importante — "esse nome vira o endereço público de vocês: github.com/seunome. Vai pro currículo. Dica: nome profissional, curto, sem números aleatórios. 'mariasilva-dev' envelhece melhor que 'mariazinha_xx99'." Se o nome desejado estiver ocupado, variações com hífen ou sobrenome resolvem.

5. Resolvam a verificação "prove que você é humano" (o quebra-cabeça).

6. O GitHub envia um código de confirmação por e-mail — abram o e-mail em outra aba, copiem o código e colem. "Não chegou? Confiram a caixa de spam."

7. Nas perguntas de personalização ("quantos membros no time?", etc.), podem pular (Skip) ou responder qualquer coisa — não muda nada.

8. No plano, escolham Free. "O plano gratuito do GitHub é absurdamente completo — vocês vão passar o curso inteiro sem pagar nada."

5.3 — Fechando o ciclo do ambiente

"Olha o que vocês têm agora: um editor profissional, uma máquina do tempo instalada e configurada com a identidade de vocês, e um portfólio público na internet com zero repositórios... por enquanto. Quando a gente conectar o VSCode ao GitHub pela primeira vez — na aula de Git — o próprio VSCode vai abrir o navegador e pedir pra vocês autorizarem com essa conta. Por isso ela precisava existir hoje."

"Ambiente pronto. Agora vem a parte boa: usar ele de verdade."

✋ Pausa

Intervalo de 10 minutos — encaixa perfeitamente aqui: fim da instalação, começo do código. Aproveite para atender individualmente quem ficou com pendência (download travado, e-mail que não chegou). Ninguém precisa do GitHub funcionando para acompanhar o resto da aula — dá pra resolver depois sem atrasar a turma.

06

Mão na massa: criando a pasta e o projeto

⏱️ ~15 min

Objetivo do bloco: todo mundo com a pasta criada, aberta no VSCode e com o arquivo index.html existindo. Ninguém avança sem isso — circule pela sala e confira as telas.

6.1 — Criando a pasta do projeto

"Regra de ouro que vocês vão levar pra vida: todo projeto vive dentro de uma pasta própria. Nunca, jamais, criem arquivo solto na Área de Trabalho ou dentro de Downloads. Projeto é como gaveta: cada um tem a sua."

⌨️ Digitar junto

Todos criam agora, junto com você:

1. Abram o Explorador de Arquivos (Windows) ou Finder (Mac).

2. Naveguem até Documentos e criem uma pasta chamada projetos (se ainda não existir).

3. Dentro dela, criem a pasta bem-cuidado.

Reforce a nomenclatura agora (previne dor de cabeça futura): nomes de pastas e arquivos sempre em minúsculas, sem espaços (use hífen: bem-cuidado, não Bem Cuidado), sem acentos ou ç. Explique: espaços e acentos quebram links e comandos no futuro — melhor criar o hábito hoje.

6.2 — Abrindo a pasta no VSCode + o conceito de pasta raiz

⌨️ Digitar junto

No VSCode: File → Open Folder (Arquivo → Abrir Pasta) e selecionem a pasta bem-cuidado. Atalho: Ctrl + K, Ctrl + O.

"Percebam uma coisa importante: a gente abriu a pasta, não um arquivo. O VSCode agora enxerga essa pasta como o universo do nosso projeto. Essa pasta principal tem um nome: pasta raiz — em inglês, root."

"Por que 'raiz'? Pensa numa árvore de cabeça pra baixo: a raiz fica no topo, e tudo cresce a partir dela. Toda subpasta — imagens, estilos, scripts — vai nascer de dentro da raiz. Quando alguém falar 'na raiz do projeto', é aqui: o primeiro nível da pasta principal."

6.3 — Criando o index.html

⌨️ Digitar junto

No painel esquerdo (Explorer), passem o mouse sobre o nome da pasta → cliquem no ícone New File (folhinha com +) → digitem index.html e Enter.

"Por que index? Convenção histórica e universal: quando você acessa um site tipo meusite.com, o servidor procura automaticamente um arquivo chamado index.html pra mostrar. Index é o 'índice', a porta de entrada. Toda página inicial de projeto se chama index — gravem isso."

"E a extensão .html diz ao computador e ao VSCode que idioma esse arquivo fala. Reparem que o ícone do arquivo já mudou — o VSCode entendeu."

✋ Pausa

Antes de codar: "Todo mundo com o index.html aberto e o nome da pasta bem-cuidado aparecendo no topo do Explorer? Levanta a mão quem travou." Resolva os travados agora — a partir daqui é digitação contínua.

07

O esqueleto do documento (e a mágica do Emmet)

⏱️ ~20 min

7.1 — Primeiro à mão, depois com atalho

Decisão pedagógica: na primeira vez, digite o esqueleto à mão com eles. Só depois apresente o atalho. Quem aprende o atalho antes do conceito nunca entende o que o atalho gera.

⌨️ Digitar junto

Linha por linha, comentando cada uma. Digite devagar — a velocidade da aula é a velocidade do aluno mais perdido:

<!-- Declara que este documento é HTML5 -->
<!doctype html>

<!-- Raiz do documento. lang="pt-BR" avisa o idioma da página -->
<html lang="pt-BR">
  <head>
    <!-- charset UTF-8: garante acentos e ç funcionando -->
    <meta charset="UTF-8" />
    <!-- viewport: faz a página se adaptar ao celular -->
    <meta name="viewport" content="width=device-width, initial-scale=1.0" />
    <!-- Texto que aparece na aba do navegador -->
    <title>Minhas Consultas · Organizador Médico</title>
  </head>
  <body>
    <!-- Tudo que o usuário vê entra aqui -->
  </body>
</html>

7.2 — Explicando linha por linha (roteiro de fala)

LinhaO que dizer para a turma
<!doctype html>"É o crachá do documento. A primeira coisa que o navegador lê. Sem ele, o navegador entra num 'modo antigo' e pode renderizar tudo torto. Não é uma tag de verdade — é uma declaração, por isso não fecha."
lang="pt-BR""Nosso primeiro atributo em ação! Ele avisa o idioma pra três públicos: leitores de tela (acessibilidade — a voz lê com sotaque certo), o Google (SEO) e o corretor do navegador."
<meta charset="UTF-8" />"Meta = informação sobre a página. UTF-8 é a tabela de caracteres que inclui ç, ã, é... Esquecer essa linha = página cheia de simbolos estranhos tipo 'GestÃ£o'. Quem nunca viu isso num site?"
<meta name="viewport" ... />"Traduzindo o content: 'a largura da página = largura do aparelho, começando com zoom de 100%'. É o que impede o site de aparecer minúsculo no celular."
<title>"O texto da abinha do navegador. Também é o que aparece de azul nos resultados do Google. Único texto 'visível' que mora no head."

7.3 — Agora sim: a mágica do Emmet

"Vocês digitaram tudo à mão, sofreram um pouquinho... agora vou contar um segredo. Apaguem TUDO. Confia."

⌨️ Digitar junto

Com o arquivo vazio, todos digitam apenas ! e apertam Tab (ou Enter na sugestão).

💥 O esqueleto inteiro aparece. Deixe o "uau" acontecer.

"Isso se chama Emmet — vem instalado no VSCode. Ele expande abreviações em HTML completo. Mas reparem num detalhe: ele gerou lang="en". O Emmet não sabe que somos brasileiros — troquem para pt-BR. Atalho nunca substitui atenção."

08

Construindo o Bem-cuidado, seção por seção

⏱️ ~60 min (coração da aula)

Método: para cada parte — (1) explique o conceito, (2) digite junto, (3) salve e veja no navegador. O feedback visual constante é o que mantém iniciante motivado. Nunca digite mais de ~10 linhas sem olhar o resultado.

8.1 — Head completo: conectando fontes externas

🖥️ Mostrar

Antes de digitar, explique o conceito de recursos externos: o HTML pode "chamar" arquivos de fora — folhas de estilo, fontes, imagens.

<!-- Link de conexão com o arquivo de estilos (criaremos na aula de CSS) -->
<link rel="stylesheet" href="styles.css" />

<!-- Importação de fonte do Google Fonts -->
<link rel="preconnect" href="https://fonts.googleapis.com" />
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin />
<link href="https://fonts.googleapis.com/css2?family=Inter:wght@400;600;700&display=swap" rel="stylesheet" />

"A tag <link> é o 'cabo' que conecta nosso HTML a outros arquivos. Dois atributos importam: rel diz a relação ('esse arquivo é uma folha de estilo') e href diz o endereço."

"E esse preconnect? Analogia: é como ligar pro restaurante avisando que você está chegando — quando você chega, a mesa já está pronta. O navegador abre a conexão com o servidor de fontes ANTES de precisar dela. Página mais rápida."

"O styles.css ainda nem existe — e tá tudo bem! O navegador procura, não acha, e segue a vida. É o nosso gancho pra aula de CSS."

8.2 — Header: o topo da página + navegação

⌨️ Digitar junto

Dentro do body:

<!-- <header> = identidade do site / saudação -->
<header class="topo">
  <div class="logo">🩺 Bem-cuidado</div>
  <nav aria-label="principal">
    <!-- Links âncora (#) levam para seções da MESMA página -->
    <a href="#proximas">Próximas</a>
    <a href="#historico">Histórico</a>
    <a href="#anexos">Exames</a>
  </nav>
</header>

Pontos de fala, linha a linha:

  • <header> — "É uma tag semântica: o nome dela já diz o que ela é. Header = cabeçalho. A gente poderia usar uma <div>? Poderia. Mas div não diz nada — é uma caixa sem etiqueta. Tags semânticas ajudam o Google e leitores de tela a entenderem a página."
  • class="topo" — "Class é um atributo de 'apelido'. Não muda nada visualmente hoje, mas é o gancho que o CSS vai usar pra estilizar. Estamos deixando pontos de encaixe prontos."
  • <nav> — "Outra semântica: navegação. E o aria-label="principal" é acessibilidade: um leitor de tela anuncia 'navegação principal' pra quem não enxerga a tela."
  • <a href="#proximas"> — "O 'a' vem de anchor, âncora. Quando o href começa com #, o link não vai pra outro site: ele rola a página até o elemento que tiver id="proximas". A gente ainda vai criar esse id — o link é a fechadura, o id é a chave."
✋ Pausa

Pergunte: "Se eu clicar em 'Histórico' agora, o que acontece?" Resposta: nada visível ainda, porque não existe elemento com id="historico". Ótimo momento pra fixar a relação href ↔ id.

8.3 — Main + a primeira section

<!-- <main> deve existir UMA única vez por página -->
<main>
  <section class="saudacao">
    <h1>Olá, Dona Lúcia 👋</h1>
    <p>Você tem <strong>2 consultas</strong> nesta semana.</p>
  </section>
</main>

"Três regras de ouro aqui. Um: <main> é o conteúdo principal e só pode existir um por página — como só existe um miolo num livro. Dois: <h1> é o título mais importante e a boa prática é ter apenas um h1 por página. Os títulos vão de h1 a h6, como capítulos e subcapítulos. Três: <strong> não é só 'negrito' — é ênfase de importância. O visual é consequência; o significado vem primeiro."

8.4 — Próximas consultas: section, article e time

⌨️ Digitar junto

Ainda dentro do main, abaixo da saudação:

<section id="proximas" aria-labelledby="t-proximas">
  <h2 id="t-proximas">Próximas consultas</h2>

  <!-- Cada consulta é um <article>: faz sentido por si só -->
  <article class="consulta">
    <time datetime="2026-07-08T14:30">Qua, 08/jul · 14h30</time>
    <h3>Dra. Marina Couto — Cardiologia</h3>
    <p>Hospital São Lucas · Sala 312</p>
    <p class="lembrete">Levar exames de sangue recentes.</p>
  </article>
</section>

Conceitos-chave para aprofundar (este é o trecho mais rico da aula):

  • section vs article vs div — "Section é um capítulo temático da página. Article é um conteúdo que faz sentido sozinho, fora do contexto: se eu recortar essa consulta e colar num e-mail, ela ainda se explica — tem data, médico, local. Isso é um article. Já a div é a caixa genérica: use quando nenhuma tag semântica servir."
  • O id apareceu! — "Lembram do link #proximas no menu? Acabamos de criar a chave. Salvem, atualizem o navegador e cliquem no link do menu. Funcionou? Isso é a dupla href/id em ação. E atenção: id é único — não podem existir dois elementos com o mesmo id na página. É como CPF."
  • aria-labelledby — "Conecta a section ao seu título pelo id. Leitor de tela anuncia: 'seção: Próximas consultas'. Acessibilidade não é extra — no nosso projeto, a Dona Lúcia pode ter baixa visão. A gente constrói pra todo mundo desde a primeira linha."
  • <time> com datetime — "Pro humano, mostramos 'Qua, 08/jul · 14h30' — bonitinho, em português. Pra máquina, o atributo datetime carrega o formato ISO: 2026-07-08T14:30, ano-mês-dia T hora. Por que os dois? Porque buscadores e leitores de tela entendem o formato padrão, e o usuário lê o formato amigável. É a mesma informação em dois idiomas: humanês e computês."
⌨️ Digitar junto

Exercício incremental (sozinhos, 3 min): "Agora criem VOCÊS o segundo article: um Ultrassom abdominal, sexta 10/jul às 09h00, na Clínica Imagem, com a observação 'Jejum de 8h'. Usem o primeiro como modelo." Depois mostre a resposta na tela para conferirem.

8.5 — Histórico: quando usar tabela

"Tabela em HTML tem má fama porque antigamente usavam pra montar o layout inteiro do site — era uma gambiarra. A regra moderna é simples: tabela serve para dados tabulares — informação que naturalmente vive em linhas e colunas, tipo planilha. Histórico de consultas com data, profissional e observação? Caso perfeito."

<section id="historico" aria-labelledby="t-historico">
  <h2 id="t-historico">Histórico médico</h2>

  <table>
    <caption>Últimas consultas realizadas</caption>
    <thead>
      <tr>
        <th scope="col">Data</th>
        <th scope="col">Profissional</th>
        <th scope="col">Observações</th>
      </tr>
    </thead>
    <tbody>
      <tr>
        <td>12/05/2026</td>
        <td>Dr. André Lima — Clínico Geral</td>
        <td>Pressão controlada. Manter medicação.</td>
      </tr>
    </tbody>
  </table>
</section>

Anatomia da tabela — a analogia da planilha:

TagTraduçãoComo explicar
<table>a planilhaA moldura de tudo
<caption>o título da planilhaDescreve a tabela — leitores de tela agradecem
<thead> / <tbody>cabeçalho / corpoSepara a linha de títulos das linhas de dados
<tr>table row = linha"Toda célula mora dentro de uma linha"
<th>table header = célula-títuloO scope="col" diz: "sou título da coluna"
<td>table data = célula de dadoO conteúdo em si
⌨️ Digitar junto

Exercício incremental (sozinhos): adicionar a segunda linha do tbody — 02/04/2026 · Dra. Helena Souza — Oftalmo · Trocar grau dos óculos. Quem terminar ajuda o colega do lado.

8.6 — Exames anexados: lista + botão

<section id="anexos" aria-labelledby="t-anexos">
  <h2 id="t-anexos">Exames anexados</h2>
  <ul class="anexos">
    <li>📄 Hemograma — 02/04/2026</li>
    <li>📄 Eletrocardiograma — 14/03/2026</li>
    <li>📄 Raio-X de tórax — 28/01/2026</li>
  </ul>
  <!-- Em um projeto futuro este botão abrirá um seletor de arquivo -->
  <button type="button">+ Anexar novo exame</button>
</section>

"Duas listas existem no HTML: <ul>, unordered list, com bolinhas — pra quando a ordem não importa, tipo lista de compras. E <ol>, ordered list, numerada — pra quando a ordem importa, tipo passo a passo de receita. Cada item, nas duas, é um <li>, list item."

"E o botão? Reparem no type="button". Botão sem type, dentro de formulário, tenta ENVIAR o formulário por padrão. Esse aqui é só visual por enquanto — na aula de JavaScript ele vai ganhar vida e abrir um seletor de arquivos. HTML é a estrutura; o comportamento vem depois."

8.7 — Formulário: a parte mais importante pro futuro

🖥️ Mostrar

Anuncie a importância: "Formulário é como o site OUVE o usuário. Login, cadastro, busca, compra — tudo é form. Prestem atenção dobrada aqui."

<section aria-labelledby="t-prefs">
  <h2 id="t-prefs">Receber lembretes por e-mail</h2>

  <form action="mailto:lembretes@bemcuidado.app" method="post" enctype="text/plain">
    <p>
      <label for="email">Seu e-mail</label><br />
      <input id="email" name="email" type="email" required
             placeholder="voce@exemplo.com" />
    </p>
    <p>
      <label for="quando">Quando avisar</label><br />
      <select id="quando" name="quando">
        <option>24 horas antes</option>
        <option>2 horas antes</option>
        <option>No dia anterior pela manhã</option>
      </select>
    </p>
    <button type="submit">Quero receber lembretes</button>
  </form>
</section>

Pontos de fala essenciais:

  • action + method — "Action é o destino: pra onde os dados vão quando o formulário for enviado. Aqui usamos mailto:, que abre o app de e-mail do computador — um truque didático, porque ainda não temos servidor. No mundo real, o action aponta pra um endereço do back-end... que VOCÊS vão construir lá na frente do curso. Method é o 'como' — falamos de GET e POST em detalhe na aula de formulários."
  • A dupla label + input — "Olha o padrão: <label for="email"> aponta pro <input id="email">. Mesmo valor nos dois. Resultado: clicar no TEXTO do rótulo já foca o campo (testem!), e o leitor de tela anuncia o rótulo certo. label for + input id = acessibilidade total. Nunca criem input sem label."
  • type="email" — "O input muda de personalidade conforme o type: email, password, date, number... Com type email, o navegador VALIDA sozinho: tentem enviar 'banana' no campo e vejam o navegador reclamar. Validação de graça, sem uma linha de JavaScript."
  • required — "Atributo booleano: a presença dele já significa 'sim, obrigatório'. Não precisa de valor."
  • name — "O id é pra página (label, CSS, âncoras). O name é a etiqueta que viaja junto com o dado quando o formulário é enviado — é como o servidor sabe qual campo é qual."
  • select + option — "Caixa de seleção: o select é a caixa, cada option é uma escolha. Quando as opções são fixas e poucas, é melhor que campo livre — ninguém digita errado."
⌨️ Digitar junto

Teste ao vivo: todos salvam, atualizam o navegador, e (1) clicam no label pra ver o foco pular pro campo, (2) tentam enviar com o campo vazio, (3) tentam enviar "banana". Deixe que descubram as mensagens de erro do navegador.

8.8 — Fechando com o footer

<footer>
  <p>Feito com 💙 pela equipe Bem-cuidado · 2026</p>
</footer>

"Footer, o rodapé — fecha o trio semântico com header e main. Reparem que ele fica FORA do main: ele não é conteúdo principal, é o fechamento da página. E... pronto. A casa da Dona Lúcia está de pé! Sem pintura, sem decoração — isso é papel do CSS — mas com uma estrutura sólida, semântica e acessível."

✋ Pausa

Pausa longa (5 min). Todos comparam a página no navegador com a sua. Circule e olhe os códigos — os erros clássicos: tag não fechada, id duplicado, elemento fora do body.

09

Analogias para fixar (use ao longo da aula)

⏱️ intercalado

Não é um bloco isolado — são cartas na manga. Saque a analogia certa no momento em que o conceito aparecer, ou use este bloco como revisão antes do encerramento.

ConceitoAnalogiaFrase pronta
HTML / CSS / JS Casa "HTML é a estrutura, CSS é a decoração, JS é a eletricidade."
Documento HTML Receita de bolo "O head é a lista de ingredientes e o forno pré-aquecido (preparação que ninguém come); o body é o modo de preparo, o passo a passo que vira bolo. E o navegador é o cozinheiro: ele lê de cima pra baixo, na ordem, e executa. Se a instrução está errada, o bolo sai errado — o cozinheiro não adivinha."
Hierarquia h1–h6 e seções Livro "h1 é o título do livro (só existe um na capa), h2 são os capítulos, h3 os subcapítulos. Ninguém escreve o capítulo 3 antes do título do livro — a hierarquia comunica importância."
Tags aninhadas Caixas dentro de caixas / matryoshka "A caixa que abriu por último fecha primeiro. Se abri section e depois article, fecho article e depois section. Caixas não se atravessam."
Atributos Etiqueta de roupa "A peça é a tag (camiseta); tamanho, cor e instruções de lavagem são os atributos."
href ↔ id Chave e fechadura "O link com # é a fechadura procurando; o id é a chave que encaixa. Sem os dois, a porta não abre."
id vs class CPF vs sobrenome "id é CPF: único, identifica UM elemento. class é sobrenome: várias pessoas podem compartilhar. Vários articles podem ter class='consulta'; só um pode ter id='proximas'."
Semântica (header, nav, main...) Potes de cozinha rotulados "Div é um pote sem rótulo: funciona, mas você precisa abrir pra saber o que tem. Tags semânticas são potes rotulados: qualquer um (Google, leitor de tela, o colega que pegar seu código) acha o açúcar de primeira."
Indentação / aninhamento Lista de tópicos e subtópicos "Quanto mais pra dentro o recuo, mais 'filho' o elemento é. É a árvore genealógica do documento — e aqui já começa a lógica de programação: hierarquia, dependência, escopo."
Ponte para a lógica: feche o bloco conectando com o futuro — "Reparem que o tempo todo a gente pensou em hierarquia (o que está dentro do quê), sequência (o navegador lê de cima pra baixo) e regras rígidas (abriu, tem que fechar). Isso É pensamento de programação. Quando chegarmos em JavaScript, vocês vão manipular exatamente essa árvore que construíram hoje — ela tem até nome: DOM."

11

Boas práticas desde o primeiro dia

⏱️ ~10 min

"Última parte teórica, e talvez a mais importante. Código funciona ou não funciona — mas código BOM é outra coisa: é código que outra pessoa entende. E essa 'outra pessoa' inclui vocês mesmos daqui a três meses, olhando pro próprio código sem lembrar de nada."

As 6 regras da turma (escreva no quadro / fixe no mural do grupo)

  1. Indentação sempre. Elemento filho entra um "degrau" (Tab) pra dentro do pai. O navegador ignora, mas o humano não: indentação é o mapa visual do aninhamento. E com Prettier + Shift Alt F, não há desculpa.
  2. Nomes de arquivo: minúsculos, sem espaço, sem acento. index.html, sobre-mim.html, styles.css. Nunca Página Inicial.html.
  3. Semântica antes de div. Antes de digitar <div>, pergunte: existe uma tag que descreve isso? Header, nav, main, section, article, footer, time...
  4. Toda tag que abre, fecha — e na ordem certa (a última que abriu é a primeira que fecha).
  5. Comente o porquê, não o óbvio. Mostre os comentários do próprio projeto: <!-- Links âncora (#) levam para seções da MESMA página --> explica uma decisão. Já <!-- parágrafo --> em cima de um <p> é ruído.
  6. Acessibilidade não é opcional. label com for, aria-label na navegação, lang no html, alt em imagens (spoiler da próxima aula). Construímos pra Dona Lúcia — e pra todo mundo.
Sobre erros: reserve 1 minuto pra normalizar o erro. "Se o código de vocês quebrou hoje, parabéns: vocês programaram. Dev sênior não é quem não erra — é quem acha o erro rápido. Esqueceu de fechar uma tag e a página inteira ficou estranha? Todo mundo aqui já passou ou vai passar por isso. O erro é o professor particular de vocês."
