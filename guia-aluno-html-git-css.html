02

Seção "Exames anexados": listas e o botão que não faz nada (ainda)

Deixe o código e o navegador lado a lado: cada trecho que você ler aqui, encontre o resultado correspondente na página.

<section id="anexos" aria-labelledby="t-anexos">
  <h2 id="t-anexos">Exames anexados</h2>

Começando pela moldura: uma section com id="anexos". Lembra do menu lá em cima, o link href="#anexos"? O id é o alvo dele — o link é a fechadura, o id é a chave. E repare num detalhe fino: o texto do link é "Exames", mas o id é "anexos". O texto visível e o id não precisam ser iguais — o que conecta os dois é o id, não a palavra.

Já o aria-labelledby="t-anexos" aponta para o id do h2 logo abaixo. Tradução: "o nome desta seção é o texto daquele título". Serve para o leitor de tela anunciar "seção: Exames anexados" a quem navega sem enxergar.

  <ul class="anexos">
    <li>📄 Hemograma — 02/04/2026</li>
    <li>📄 Eletrocardiograma — 14/03/2026</li>
    <li>📄 Raio-X de tórax — 28/01/2026</li>
  </ul>

ul vem de unordered list: lista sem ordem, aquela de bolinhas. A pergunta que define qual lista usar é: a ordem importa? Lista de compras — tanto faz se o arroz vem antes do feijão — é ul. Modo de preparo de uma receita — se você assar antes de misturar, vira desastre — é ol, ordered list, numerada. Aqui são exames: a ordem não muda nada, então ul.

Cada item, independente da lista, é um li — list item. E o emoji 📄 é conteúdo comum, como qualquer letra: para o HTML, emoji é só mais um caractere. É o UTF-8 do nosso meta charset garantindo que ele apareça.

✏️ Pratique

Experimento de 1 minuto: troque <ul> por <ol> (abertura e fechamento), salve e olhe o navegador — as bolinhas viram 1, 2, 3. Depois desfaça com Ctrl + Z duas vezes e salve de novo. Percebe como testar hipótese é barato? Mudou, salvou, olhou, desfez. Faça isso o tempo todo.

  <button type="button">+ Anexar novo exame</button>
</section>

Agora o botão. button é clicável por natureza — o navegador já dá aparência de botão, efeito de clique e foco pelo teclado. Mas preste atenção no type="button". Botão tem três tipos: submit (envia formulário), reset (limpa formulário) e button (não faz nada sozinho). Este aqui é o terceiro tipo: uma campainha instalada, mas ainda sem fio ligado. O toque — abrir a janelinha de escolher arquivo — quem vai ligar é o JavaScript, daqui a algumas aulas.

Por que já colocar o botão então? Porque HTML é a estrutura: deixamos a campainha na parede, bonitinha, no lugar certo. A eletricidade vem depois. Clique nele no navegador: nada acontece — e está certo assim.

🧠 Teste-se

Se esse botão estivesse DENTRO de um formulário e sem o atributo type, o que aconteceria ao clicar?

Ver resposta

Dentro de um form, botão sem type age como submit e tenta enviar o formulário. Por isso declarar o type sempre é boa prática.

03

Seção "Mídias", parte 1: imagens

Nova seção no projeto: id="midias" — e repare que o menu no topo ganhou um quarto link, #midias, seguindo exatamente o padrão que você já conhece. Aqui a página ganha vida: imagem, vídeo e áudio. Vamos por partes, começando pela mídia mais comum da internet inteira: a imagem.

<img
  src="imagens/foto-clinica.png"
  alt="Fachada da clínica médica parceira"
  width="300"
  height="200"
/>

src: o endereço — e a diferença entre relativo e absoluto

Primeira coisa: img é uma tag vazia — não tem conteúdo dentro, não tem fechamento. Ela É a imagem. O atributo src, de source (fonte), diz onde a imagem mora.

Esse primeiro src é um caminho relativo: imagens/foto-clinica.png. Traduzindo: "a partir de onde EU estou" — e o "eu" é o arquivo HTML — "entre na pasta imagens e pegue o arquivo foto-clinica.png". É como dar direção para alguém que está do seu lado: "vira na segunda rua à direita". Só funciona a partir de onde a pessoa está.

Agora olhe a segunda imagem do código:

<img
  src="https://images.unsplash.com/photo-1576091160399-112ba8d25d1d?w=300"
  alt="Médico utilizando estetoscópio"
/>

Este é um caminho absoluto: começa com https://, é o endereço completo na internet. É o CEP com rua, número, cidade e país — funciona de qualquer lugar do mundo, porque a imagem vem de um servidor externo, o Unsplash. Regra de bolso: arquivo seu, na sua pasta → relativo. Arquivo da internet → absoluto.

📌 Nota

Curiosidade nerd: o ?w=300 no fim da URL é um pedido ao servidor — "me manda essa foto com 300 pixels de largura". Você vai ver muito esses "?" em URLs; são parâmetros, conversas com o servidor.

alt: o atributo que separa amador de profissional

O alt é o texto alternativo — e aqui vale ser direto: imagem sem alt é imagem errada. Ele serve para duas situações. Um: a imagem não carregou — internet caiu, arquivo sumiu — e o navegador mostra o texto no lugar. Dois, e mais importante: a Dona Lúcia pode ter baixa visão e usar leitor de tela. Sem alt, o leitor fala "imagem"... e só. Com alt, ele fala "Fachada da clínica médica parceira". É a diferença entre incluir e excluir alguém do seu site.

Dica de escrita: descreva o que a imagem COMUNICA, como se estivesse contando por telefone. "foto1.png" não é alt. "Fachada da clínica" é.

width e height: reservando a vaga

Por fim, width="300" height="200" — largura e altura em pixels. Além de dimensionar, há um motivo escondido: quando o navegador conhece o tamanho ANTES de baixar a imagem, ele reserva o espaço na página. Sem isso, a página carrega, você começa a ler, a imagem chega atrasada e empurra todo o texto para baixo — aquele "pulo" irritante que faz você clicar no lugar errado. Declarar o tamanho é reservar a vaga do estacionamento antes de o carro chegar.

✏️ Pratique

Experimento do alt (2 min): quebre o src da primeira imagem de propósito — troque para imagens/foto-clinicaaa.png — salve e olhe o navegador: aparece o ícone de imagem quebrada + o texto do alt. Conserte em seguida.

Aliás: a imagem local só aparece se você criou a pasta imagens com o arquivo dentro. Se não tem, você está vendo o alt agora — e isso é o alt funcionando, não um erro. Já a do Unsplash aparece para todo mundo: ela vem da internet.

🧠 Teste-se

Se você mover o index.html para outra pasta e deixar a pasta imagens para trás, qual das duas imagens quebra?

Ver resposta

A local (caminho relativo) quebra, porque o "a partir de onde eu estou" mudou. A do Unsplash (caminho absoluto) continua funcionando de qualquer lugar.

04

Seção "Mídias", parte 2: vídeo, o caso YouTube e áudio

4.1 — O player de vídeo nativo

<video controls width="400">
  <source src="videos/tutorial-mapa.mp4" type="video/mp4" />
  Seu navegador não suporta a reprodução de vídeo.
</video>

O <video> coloca um player de vídeo direto na página, sem instalar nada. Três coisas para dissecar:

Primeira: o atributo controls. Repare que ele está sozinho, sem ="alguma coisa". É um atributo booleano — a simples presença dele significa "sim". Com controls, aparecem play, pause, volume, barrinha de progresso. Sem ele? O vídeo fica ali, mudo e parado, sem nenhum botão. É um interruptor: está na parede ou não está.

Segunda: o <source> lá dentro. Ele diz de onde vem a mídia — src relativo, pasta videos, igualzinho à imagem local. E o type="video/mp4" é o chamado MIME type: a etiqueta na caixa antes de abrir. O navegador lê "video/mp4" e já sabe se consegue reproduzir ANTES de gastar internet baixando. Dá até para colocar vários source com formatos diferentes — o navegador usa o primeiro que entender.

Terceira: o texto solto, "Seu navegador não suporta...". Isso é o fallback, o plano B: só aparece em navegadores tão antigos que nem conhecem a tag video. É rede de segurança — provavelmente você nunca vai vê-lo, e ótimo.

4.2 — O mistério do YouTube

✏️ Pratique

Antes de ler a explicação, faça o teste: troque o src do segundo vídeo por uma URL do YouTube (https://www.youtube.com/embed/...), salve e olhe: player preto, quebrado. Pergunta: por que não funcionou, se a URL está certa? Pense 1 minuto antes de continuar lendo.

A resposta: o <source> precisa apontar para um arquivo de vídeo — um .mp4, .webm. E aquela URL do YouTube não devolve um arquivo de vídeo: devolve uma PÁGINA — o player do YouTube, que é HTML. O navegador tenta decodificar uma página como se fosse filme, obviamente não consegue, e trava no preto. O YouTube esconde o arquivo bruto de propósito: é assim que eles garantem anúncios e direitos autorais.

Então como colocar YouTube no site? Com outra tag:

<iframe
  width="400" height="225"
  src="https://www.youtube.com/embed/kbgtymfFL4o"
  title="Orientações gerais de saúde"
  allowfullscreen
></iframe>

O <iframe> abre uma janela para outro site dentro do seu — uma TV dentro da TV. Você não está tocando o vídeo: está incorporando a página do player do YouTube inteira, com tudo que ela sabe fazer. A URL "embed" o próprio YouTube fornece: botão Compartilhar → Incorporar.

Três diferenças em relação ao video que valem prova: iframe não usa source nem type (o src vai direto nele); iframe precisa de fechamento — </iframe> — não é tag vazia; e o title aqui é acessibilidade: descreve o conteúdo da janela para o leitor de tela. O allowfullscreen? Outro booleano — libera o botão de tela cheia.

4.3 — Áudio: o irmão sem imagem do vídeo

<audio controls>
  <source src="https://www.w3schools.com/html/horse.mp3" type="audio/mpeg" />
  Seu navegador não suporta a reprodução de áudio.
</audio>

Se você entendeu o video, o <audio> vem de graça: mesma lógica — controls booleano, source com src e type, texto de fallback. A diferença é que não tem imagem: só a barrinha de controle. No projeto temos dois: a mensagem de boas-vindas local (pasta audios) e um da internet.

📌 Nota

Pegadinha clássica de prova e de entrevista: o MIME type do MP3 é audio/mpeg — não existe "audio/mp3". MP3 é apelido; o sobrenome oficial do formato é MPEG. Anote essa.

✏️ Pratique

Dê play no áudio da internet (o horse.mp3 — sim, é um cavalo relinchando, pode rir). Depois: apague a palavra controls, salve, atualize — o player some por completo. Volte com o controls. Booleano: presente = existe, ausente = não existe. Não tem meio termo.

🧠 Teste-se

Quiz de 30 segundos: video usa source? E iframe? Qual dos dois fecha com tag de fechamento? Qual o MIME type do MP3?

Ver respostas

Video usa source (pode ter vários). Iframe não — o src vai direto nele. Os dois fecham com tag de fechamento, mas por motivos diferentes: video envolve sources e fallback; iframe simplesmente não é tag vazia. E o MIME type do MP3 é audio/mpeg.

05

Formulário: como o site ouve o usuário (+ footer)

Última seção da página — e a mais importante para o seu futuro. Tudo que é interação séria na web é formulário: login, cadastro, busca, compra, comentário. Atenção dobrada nesta parte.

<form
  action="mailto:lembretes@bemcuidado.app"
  method="post"
  enctype="text/plain"
>

O <form> é o envelope: agrupa campos que serão enviados JUNTOS. O action é o destinatário escrito no envelope — para onde os dados vão. Aqui usamos mailto:, que abre o programa de e-mail do usuário: um truque didático, porque ainda não temos servidor. Em site de verdade, o action aponta para o back-end — que é o destino final deste curso, aliás. O method="post" é COMO o envelope viaja: post manda os dados escondidos no corpo do envio; o outro método, get, cola tudo na URL, à vista de todos. Senha, por exemplo, JAMAIS viaja por get. E o enctype="text/plain" é o formato do conteúdo — texto simples, necessário para o mailto ficar legível.

  <label for="email">Seu e-mail</label><br />
  <input
    id="email" name="email" type="email"
    required placeholder="voce@exemplo.com"
  />

A dupla sagrada: label for="email" apontando para o input id="email". Mesmo valor nos dois = rótulo grudado no campo. Teste depois: clicar no TEXTO "Seu e-mail" já acende o campo. E para o leitor de tela, o campo tem nome. Nunca crie input sem label.

Agora o input, atributo por atributo:

  • id — a identidade do campo DENTRO da página: é o que o label (e futuramente o CSS e o JavaScript) enxergam.
  • name — a etiqueta do dado QUANDO VIAJA: o envio vira "email=valor". Guarde a diferença assim: id é o nome em casa, name é o nome no crachá do envio. Detalhe cruel: campo sem name simplesmente não é enviado. Some. É o bug clássico de iniciante.
  • type="email" — o input é um camaleão: muda de personalidade pelo type. Com email, o celular abre o teclado com @ e o navegador valida o formato sozinho.
  • required — booleano de novo! Presença = campo obrigatório: o navegador bloqueia envio vazio.
  • placeholder — o texto cinza de exemplo que some quando você digita. Atenção: ele NÃO substitui o label — exemplo é exemplo, rótulo é rótulo.

E o <br />: quebra de linha forçada, tag vazia. Joga o input para a linha de baixo do label. Funciona, mas use com moderação — espaçamento bonito é trabalho para o CSS.

  <select id="quando" name="quando">
    <option>24 horas antes</option>
    <option>2 horas antes</option>
    <option>No dia anterior pela manhã</option>
  </select>

  <button type="submit">Quero receber lembretes</button>
</form>

O select é o menu suspenso — a caixinha que abre opções. Cada option é uma escolha. Repare que as options não têm atributo value: nesse caso, o próprio texto vira o valor enviado. Quando as respostas possíveis são poucas e fixas, select é melhor que campo livre: ninguém digita "24hrs antes" de um jeito e "24 horas" de outro.

E feche o ciclo: ESTE botão é type="submit". Compare com o da seção de exames, que era type="button" e não fazia nada. O submit dispara a validação (o required entra em ação) e executa o action. Mesma tag, papéis opostos — o type é quem decide.

✏️ Pratique

Bateria de testes no navegador (3 min): (1) clique no texto do label → o campo foca; (2) envie com o campo vazio → o navegador bloqueia; (3) digite "banana" e envie → erro de formato; (4) e-mail válido + enviar → o programa de e-mail tenta abrir. Repare: três validações e você não escreveu UMA linha de JavaScript. HTML bem feito trabalha de graça para você.

E o fim da página: <footer>, o rodapé semântico — créditos, copyright, links institucionais — fechando o trio com header e main. Página completa: estrutura, mídia, interação. A casa está pronta e mobiliada... só falta a pintura, que vem na Parte 3. Mas antes: como três pessoas constroem essa casa juntas sem uma derrubar a parede da outra?

Parte 2 · Git e colaboração em equipe

Seu primeiro fluxo de trabalho profissional — leia e execute com o seu trio

06

Git sem medo: o que é e por que você vai amar

Você já fez trabalho em grupo mandando arquivo pelo WhatsApp? "TCC_final.docx", "TCC_final_v2.docx", "TCC_final_v2_AGORA_VAI.docx"... e no fim alguém edita a versão errada e perde-se uma noite de trabalho. Todo mundo já viveu isso. O Git existe para isso nunca mais acontecer.

O Git é um sistema de controle de versões: ele guarda o histórico completo do projeto — cada "foto" que alguém tirou, quem tirou, quando e por quê. E o GitHub é o lugar na nuvem onde esse histórico fica guardado e compartilhado entre o time. Na analogia que você já conhece: Git é a câmera, GitHub é o álbum online.

E respondendo com clareza a pergunta da abertura: a entrega do trabalho é UM único repositório por trio — um projeto só, com o histórico mostrando a contribuição das três pessoas. Não é "cada um faz o seu e alguém cola tudo junto no final". É todo mundo construindo no mesmo lugar, ao mesmo tempo, com segurança. Parece mágica, é método — e nas próximas seções você vai dominar esse método.

📌 Nota

Quer uma prova de que funciona? Abra qualquer projeto famoso de código aberto no GitHub e olhe a aba de histórico de commits: nomes, datas, mensagens. Milhares de pessoas, um repositório, ninguém apagou o trabalho de ninguém. Se funciona para elas, funciona para o seu trio.

📌 Nota

Preparação do trio: UM integrante cria o repositório no GitHub (Repositories → New → nome combinado, ex.: bem-cuidado-equipe → Public → Add README) e adiciona os outros dois em Settings → Collaborators. Os convidados aceitam o convite que chega por e-mail. Só então sigam para o clone.

07

git clone: trazendo o projeto para a sua máquina

Primeiro passo de qualquer trabalho em equipe: todo mundo precisa de uma cópia do projeto. Isso é o git clone. Ele baixa a pasta inteira do projeto — arquivos E histórico completo — para a sua máquina. E o detalhe que dá nome à mágica: a cópia continua conectada ao repositório original lá no GitHub, que chamamos de remoto. Não é um download morto: é um clone vivo, que sabe de onde veio e consegue mandar e receber novidades.

✏️ Pratique

1. No GitHub, no repositório do trio: botão verde Code → aba HTTPS → ícone de copiar a URL.

2. No VSCode, abra o terminal (Ctrl + ') e navegue até a sua pasta de projetos: cd Documents/projetos (ou onde você guarda os seus).

3. Rode o clone (colar no terminal: Ctrl + Shift + V ou botão direito):

git clone https://github.com/usuario-do-trio/bem-cuidado-equipe.git
✏️ Pratique

4. Entre na pasta recém-criada e abra no VSCode:

cd bem-cuidado-equipe
code .

O code . — "code ponto" — significa "abra o VSCode AQUI, nesta pasta". O ponto, no terminal, é sempre "a pasta onde estou". Pronto: cada um dos três agora tem uma cópia completa e conectada do projeto. Na primeira vez, o GitHub vai pedir login — o VSCode abre o navegador, você autoriza com a sua conta, e nunca mais pede.

📌 Nota

Erro comum: clonar dentro de outro repositório já clonado (repositório dentro de repositório). Confirme que você está na sua pasta de projetos antes do clone — o pwd no terminal mostra onde você está.

08

Branches: cada um trabalhando no seu universo paralelo

Agora o conceito mais importante da colaboração: a branch — "ramo", em inglês. Pense no projeto como um documento oficial publicado: a linha do tempo principal, que se chama main, é a versão limpa, que funciona, que qualquer um pode abrir com confiança. Regra número um do trabalho em equipe: ninguém rabisca o documento oficial diretamente.

Em vez disso, cada pessoa cria a sua branch: uma cópia paralela da linha do tempo, um universo alternativo onde você pode experimentar, errar, refazer — sem encostar na main e sem encostar no universo dos colegas. A Ana mexe na seção de mídias na branch dela, o Bruno melhora a tabela na dele, a Carla cuida do formulário na dela. Três universos, zero conflito. Quando o trabalho de alguém fica pronto e aprovado, aí sim ele é incorporado de volta à main — isso é o merge, que vem daqui a pouco.

✏️ Pratique

Crie a SUA branch (cada integrante com um nome diferente!). O comando cria e já muda para ela:

git checkout -b ana/secao-midias

Dissecando: checkout = mudar de branch; -b = "e crie ela antes, porque não existe ainda". E o nome: adote o padrão seunome/o-que-vou-fazer — "ana/secao-midias", "bruno/tabela-historico". Um nome de branch bom conta a história sozinho: quem e o quê.

✏️ Pratique

Dois comandos de orientação — decore, você vai usar o tempo todo:

git branch          # lista as branches; o * marca onde você está
git status          # o "onde estou e o que mudou?" — o comando mais digitado do mundo
📌 Nota

Olhe também o canto inferior esquerdo do VSCode: o nome da branch atual fica sempre visível ali, e dá para trocar de branch clicando. Olhe para esse cantinho SEMPRE antes de começar a editar. "Em que branch eu estou?" é o cinto de segurança do Git.

09

add, commit e push: salvando e publicando o trabalho

Este é o ciclo que você vai repetir mil vezes na carreira. A analogia é uma foto de família: o git add escolhe quem entra na foto — quais arquivos modificados serão registrados. O git commit tira a foto e escreve a legenda — cria o registro permanente no histórico, com mensagem, autor e data. E o git push manda a foto para o álbum da família na nuvem — publica seus commits no GitHub, onde o time vê.

Por que o add existe separado do commit? Porque nem tudo que você mexeu precisa entrar na mesma foto. O add é a área de preparação — em inglês, staging — onde você monta a cena antes do clique.

✏️ Pratique

Mão na massa com mudança real: edite algo pequeno no index.html (um texto, um comentário com o seu nome). Depois, o ciclo completo:

git status                                    # 1. o que mudou? (arquivo aparece em vermelho)
git add index.html                            # 2. escala o arquivo pra foto (dica: git add . adiciona tudo)
git status                                    # 3. confira: agora está verde, pronto pra foto
git commit -m "Adiciona seção de mídias com vídeo e áudio"   # 4. clique! foto tirada com legenda
git push -u origin ana/secao-midias           # 5. publica a branch no GitHub

Sobre a mensagem do commit, o -m: escreva o que a mudança FAZ, no presente — "Adiciona seção de mídias", "Corrige link do menu". Mensagens como "mudanças", "aaaa" ou "agora vai" são o equivalente a legendar todas as fotos do álbum com "foto". Daqui a um mês, a mensagem é a única memória do que aconteceu.

Sobre o push: nessa PRIMEIRA vez, vai o -u origin nome-da-branch — é a apresentação formal: "GitHub, essa minha branch nova corresponde a essa daí". origin é o apelido padrão do repositório remoto. Das próximas vezes, um git push seco resolve. Se esquecer, relaxe: o próprio terminal imprime o comando certo para copiar.

✏️ Pratique

Atualize a página do repositório no GitHub: a sua branch apareceu, com o aviso amarelo "Compare & pull request". Seu trabalho está na nuvem, visível para o time — mas repare: a main continua intocada. Esse botão amarelo é exatamente o próximo capítulo.

🧠 Teste-se

Coloque em ordem: commit, push, add, status. E o que acontece se você commitar sem ter dado add?

Ver resposta

status (ver o que mudou) → add (preparar) → commit (registrar) → push (publicar). Commit sem add não registra nada novo — o Git avisa que não há mudanças preparadas ("nothing added to commit"). Leia a mensagem do git status com calma: ele literalmente diz o que fazer.

10

Pull Request e merge: juntando tudo com segurança

O trabalho está na sua branch, publicado. Como ele entra na main? Pela porta da frente: o Pull Request — PR para os íntimos. Traduzindo o nome esquisito: "pedido para puxarem minhas mudanças". É você dizendo para o time: "terminei minha parte, olhem o que eu fiz, e se estiver bom, juntem à versão oficial".

E aqui mora o segredo dos times profissionais: o PR não é burocracia, é revisão. Antes de qualquer coisa entrar na main, pelo menos um colega olha, comenta, aprova. Quatro olhos erram menos que dois. No seu trio, a regra é: quem abre o PR não aprova o próprio PR — sempre outro integrante revisa.

✏️ Pratique

Fluxo completo no GitHub, com um PR de verdade:

1. No repositório, clique no aviso "Compare & pull request" (ou aba Pull requests → New).

2. Confira a direção da seta: base: main ← compare: sua-branch — "quero levar minha branch PARA a main".

3. Título claro + descrição curta: o que foi feito e como testar. Create pull request.

4. Outro integrante abre o PR, clica na aba Files changed — verde = adicionado, vermelho = removido — e deixa um comentário (elogio vale!).

5. Aprovado? Botão Merge pull request → Confirm. A main acabou de receber a contribuição, com autoria registrada.

Depois do merge, um passo que separa amador de profissional: todo mundo sincroniza. A main mudou lá na nuvem; as cópias locais ainda não sabem. Cada integrante roda:

git checkout main       # volta pra linha do tempo oficial
git pull                # puxa as novidades do GitHub pra sua máquina

O git pull é o irmão gêmeo do push, na direção contrária: push empurra para a nuvem, pull puxa da nuvem. Ritual sagrado do trabalho em equipe: começou o dia? git pull na main ANTES de criar branch nova. Trabalhar em cima de versão velha é a receita número um de conflito.

11

Por que ninguém sobrescreve o trabalho de ninguém

Hora de responder com todas as letras a pergunta do começo. Três motivos pelos quais o Git torna quase impossível uma pessoa apagar o trabalho da outra sem querer:

Um: o Git soma, não substitui. Quando dois trabalhos são juntados no merge, o Git não pega o arquivo de um e joga fora o do outro. Ele compara linha por linha e junta: a Ana mexeu na seção de mídias, o Bruno na tabela — linhas diferentes, o Git costura os dois automaticamente e ninguém perde nada. Pense num editor de livro juntando capítulos de três autores: cada capítulo entra inteiro no livro final.

Dois: quando há dúvida, o Git PARA e pergunta. E se duas pessoas mexerem na mesma linha do mesmo arquivo? Aí é o famoso conflito — e olha que interessante: conflito não é o Git quebrando, é o Git sendo honesto. Ele se recusa a escolher sozinho, marca o trecho com as duas versões lado a lado e diz: "humanos, decidam". Vocês olham, escolhem (ou combinam as duas), commitam a decisão. Nada é descartado em silêncio, nunca. No VSCode, o trecho conflitado aparece marcado com <<<<<<< / ======= / >>>>>>> e botões prontos: "Accept Current", "Accept Incoming", "Accept Both".

Três: mesmo o "apagado" não morre. Cada commit fica gravado no histórico para sempre. Alguém deletou uma seção por engano três commits atrás? O histórico tem a foto antiga — dá para recuperar. No Git, desastre de verdade é raríssimo; o que existe é bagunça temporária, e bagunça se arruma.

✏️ Pratique

Experimento avançado (opcional, mas vale ouro): provoque um mini-conflito de propósito com um colega. Os dois editam a MESMA linha do h1 com textos diferentes, cada um na sua branch; façam merge de uma, depois tentem a outra — e resolvam o conflito juntos no VSCode. Resolver um conflito uma vez, com calma, tira 90% do medo para sempre.

12

Boas práticas do trio

O ciclo diário do time (fixe no grupo do trio)

git pull (main) → git checkout -b nome/tarefa → editar → git add → git commit -m → git push → Pull Request → colega revisa → Merge → todos: git pull

As 7 regras de ouro da colaboração

  1. Dividam o trabalho por seções/arquivos diferentes e combinem ANTES quem faz o quê — a maior parte dos conflitos nasce de falta de conversa, não de falta de técnica. Sugestão para o Bem-cuidado: pessoa A = seção de mídias, pessoa B = tabela + anexos, pessoa C = formulário + footer.
  2. Nunca trabalhe direto na main. Branch sempre, mesmo para mudança pequena.
  3. Commits pequenos e frequentes — várias fotos nítidas contam a história melhor que uma foto borrada de tudo. Terminou um pedacinho que funciona? Commit.
  4. Mensagens de commit que expliquem — "Adiciona player de áudio na seção de mídias", nunca "mudanças".
  5. git pull na main antes de começar qualquer dia de trabalho e antes de criar branch nova.
  6. PR revisado por outro integrante, sempre — revisar também ensina: lendo o código do colega, você aprende o dobro.
  7. Comuniquem-se fora do Git também — "vou mexer no header agora" no grupo do WhatsApp evita duas pessoas na mesma linha.

Recapitulando a entrega: um repositório por trio, com a main funcionando e o histórico mostrando commits e PRs das três pessoas — é assim que a contribuição de cada um será avaliada, então nada de uma pessoa commitar pelo grupo todo. E respire: você acabou de aprender o fluxo de trabalho usado em basicamente toda empresa de tecnologia do planeta. O que muda de você para um time do Google é o tamanho do projeto, não o processo.

A casa está construída e mobiliada; chegou o dia da pintura, da decoração, do acabamento: CSS.

Parte 3 · Introdução ao CSS

Estude com o styles.css aberto ao lado do index.html — leia, digite, salve, olhe

C1

O que é CSS — e por que ele existe

CSS: Cascading Style Sheets — Folhas de Estilo em Cascata. Traduzindo: "folhas de estilo" porque é onde escrevemos as regras de aparência — cores, fontes, tamanhos, espaçamentos. E "em cascata" porque as regras escorrem de cima para baixo, como uma cachoeira — uma regra mais específica, mais abaixo, pode refinar uma mais geral. Isso vai fazer sentido na prática.

A analogia é a mesma da nossa casa: o HTML é o esqueleto; o CSS é a roupa e a maquiagem. Olhe a sua página agora: Times New Roman, fundo branco, tudo alinhado à esquerda... É um esqueleto saudável, mas pelado. Hora de vesti-lo.

Os 5 motivos para usar CSS, com exemplo real em cada um

  • Separar conteúdo do design — o texto da consulta da Dona Lúcia (conteúdo) não deve se misturar com "a cor do lembrete é amarela" (design). Assuntos separados, arquivos separados: quem escreve o conteúdo não precisa entender do visual e vice-versa.
  • Manter consistência visual — imagine o site da clínica com 200 páginas, todos os títulos azuis. Se cada página tivesse sua própria cor escrita à mão... uma hora alguém erraria o tom de azul. Com CSS, a regra é escrita UMA vez e vale para todas.
  • Facilitar a manutenção — o chefe pede: "muda o azul pra verde". Sem CSS: editar 200 páginas. Com CSS: editar 1 linha. Fim do expediente.
  • Criar layouts responsivos — é o CSS que faz o site se reorganizar no celular. Lembra do meta viewport que preparamos no HTML? Ele avisa o navegador; o CSS faz a mudança acontecer.
  • Melhorar a experiência do usuário — texto legível, botão que parece botão, espaçamento que deixa respirar. A Dona Lúcia precisa achar a consulta dela em 5 segundos: isso é design a serviço da pessoa.
✏️ Pratique

Experimento que vale mil palavras: abra qualquer site conhecido, aperte F12 → aba Elements → e desmarque regras de estilo do body (as caixinhas de checkbox no painel Styles). O site vai virando uma lista pelada de textos e links. ISSO é a web sem CSS — tudo que você acha bonito na internet é CSS trabalhando.

C2

Sintaxe: seletor, propriedade e valor

h1 {
  color: blue;
  font-size: 24px;
}

Toda regra CSS do planeta tem essa cara — decorou essa estrutura, decorou o CSS inteiro. Três pedaços:

  • O seletor — o h1 antes das chaves — responde QUEM vai ser estilizado. É o destinatário do envelope.
  • A propriedade — color, font-size — responde O QUE vai mudar. A característica.
  • O valor — blue, 24px — responde COMO fica. E cada dupla propriedade-valor termina com ponto e vírgula, como um ponto final de frase.

Analogia de ordem para o chefe de cozinha: "Nos bolos de chocolate {cobertura: morango; tamanho: grande;}". Quem recebe a ordem? Todos os bolos de chocolate. O quê muda? Cobertura e tamanho. Como? Morango e grande. CSS é isso: ordens de aparência para grupos de elementos.

📌 Nota

Os símbolos, porque errar símbolo é 90% dos bugs de iniciante: chaves { } abraçam as ordens; dois-pontos separam propriedade de valor; ponto e vírgula encerra cada ordem. Esqueceu um ponto e vírgula? A regra seguinte para de funcionar em silêncio — e você fica meia hora procurando.

✏️ Pratique

Sua primeira regra CSS, no próprio Bem-cuidado — dentro da tag <style> que está esperando vazia no head desde a primeira aula:

h1 {
  color: darkblue;
  font-size: 40px;
}
✏️ Pratique

Salve → olhe o navegador: o "Olá, Dona Lúcia" mudou de cor e tamanho. Agora, playground livre: troque o valor de color (red, green, tomato, rebeccapurple...) e o font-size. Errou o nome da cor? O navegador ignora a linha e segue. CSS não trava a página — no máximo, ele te ignora.

🧠 Teste-se

Na regra que você escreveu: quem é o seletor? Quantas propriedades tem? O que acontece se apagar o ponto e vírgula do color?

Ver resposta

O seletor é h1. Há duas propriedades: color e font-size. Sem o ponto e vírgula do color, o navegador se confunde e a regra do font-size deixa de funcionar — silenciosamente.

C3

As 3 formas de vincular CSS ao HTML

Vamos da pior para a melhor, com a analogia do guarda-roupa:

1. CSS inline — o style escrito dentro da própria tag, <h1 style="color:red"> — é costurar a etiqueta de lavagem em cada peça de roupa, uma por uma. Funciona? Funciona. Escala? Nunca. Para mudar tudo, você recostura peça por peça. Evite.

2. CSS interno — dentro da tag <style> no head — é deixar as roupas dobradas no quarto: organizado, mas só serve para ESTE quarto, esta página. Foi o que você usou agora há pouco, e é ótimo para aprender e testar.

3. CSS externo — um arquivo .css separado, conectado pelo <link> — é o closet profissional: todas as páginas do site se vestem no mesmo lugar. Uma mudança no closet, o site inteiro muda de figurino. É o recomendado e o padrão do mercado.

📌 Nota

Momento em que o passado faz sentido: abra o head do Bem-cuidado. A linha <link rel="stylesheet" href="styles.css" /> está lá desde o PRIMEIRO dia de HTML, apontando para um arquivo que nunca existiu. Deixamos o cabide pronto esperando a roupa. Hoje a roupa chegou.

✏️ Pratique

Migração oficial:

1. Crie o arquivo styles.css na mesma pasta do index.html (o href é relativo — mesma regra dos caminhos de imagem!).

2. Recorte a regra do h1 de dentro do <style> e cole no styles.css. Detalhe: no arquivo .css não existe tag style — é CSS puro, direto. E comentário lá é /* assim */, não <!-- assim -->.

3. Salve os dois arquivos → navegador: tudo continua funcionando, agora vindo do arquivo externo. A tag style pode ficar vazia ou ser removida.

C4

Propriedades essenciais: o kit de sobrevivência

Quatro grupos de propriedades resolvem a maior parte do dia a dia. Estude cada grupo e aplique IMEDIATAMENTE no projeto — teoria e prática intercaladas, nunca quatro grupos seguidos sem codar.

Grupo 1 — Cor e fundo

color é a cor do TEXTO — pegadinha do idioma: não é "cor de tudo", é cor da letra. background-color é a cor de FUNDO do elemento, a parede atrás do texto.

✏️ Pratique

No styles.css:

body {
  background-color: #f5f7fa;
}

.lembrete {
  background-color: lightyellow;
  color: darkred;
}

Duas novidades aí. O #f5f7fa é uma cor em código hexadecimal — o "CPF da cor": # + seis caracteres que descrevem a mistura exata de vermelho, verde e azul. Nomes como blue são apelidos limitados; o hex dá QUALQUER cor. Não decore códigos: todo seletor de cor gera eles para você. E o .lembrete com ponto na frente... é o gancho da class que deixamos no HTML — a seção C6 explica direitinho. Olhe o navegador: o aviso "Levar exames de sangue" virou um alerta de verdade.

Grupo 2 — Tipografia

✏️ Pratique

Amplie a regra do body:

body {
  background-color: #f5f7fa;
  font-family: 'Inter', Arial, sans-serif;
  font-size: 16px;
}

font-family escolhe a fonte — e olha o momento: 'Inter' é a fonte que importamos do Google Fonts lá na primeira aula de HTML. O link estava no head esperando; agora ela entra em cena. A lista com vírgulas é o plano B do plano B: se a Inter falhar, usa Arial; se nem Arial, qualquer fonte "sans-serif" — sem serifa, aquelas letras retinhas sem pezinho. font-size é o tamanho, e font-weight o peso: 400 normal, 700 negrito — os mesmos números 400;600;700 da URL do Google Fonts, não é coincidência.

Grupos 3 e 4 — Espaçamento e bordas (a ponte para o Box Model)

✏️ Pratique

A regra que transforma as consultas em cartões:

.consulta {
  background-color: white;
  border: 1px solid #cccccc;
  padding: 16px;
  margin: 12px;
}

border é a moldura: três valores numa linha — espessura (1px), estilo (solid; existem dashed, dotted...) e cor. E agora as duas palavras que confundem TODO iniciante do mundo: padding e margin. As duas criam espaço — mas de lados diferentes da borda. Segure essa dúvida por um minuto: a próxima seção existe exatamente para resolvê-la.

C5

O Box Model: todo elemento é uma caixa

Se você só levar UMA ideia desta parte para casa, que seja esta: para o navegador, todo elemento HTML é uma caixa retangular. Título, parágrafo, imagem, botão — tudo caixa. Não existe círculo, não existe forma livre: caixas, caixas, caixas. Aprender CSS de layout é aprender a organizar caixas.

E toda caixa tem quatro camadas. Analogia do quadro na parede:

  • O conteúdo é a pintura em si — o texto, a imagem.
  • O padding é o passe-partout, aquela margem interna branca entre a pintura e a moldura — espaço DENTRO da caixa.
  • A border é a moldura — a linha que você já desenhou nos cartões.
  • A margin é a distância entre um quadro e o quadro vizinho na parede — espaço FORA da caixa, que afasta os elementos entre si.
📌 Nota

Macete definitivo para nunca mais confundir: Padding = Por dentro. Margin = "Mora fora". Padding empurra a borda para longe do conteúdo; margin empurra os VIZINHOS para longe da borda.

✏️ Pratique

Experimento de contraste (o que fixa o conceito de verdade): na regra .consulta, mude padding: 16px para padding: 50px → salve → o cartão INCHA, o texto ganha respiro interno. Volte para 16px. Agora margin: 12px para margin: 50px → os cartões se AFASTAM um do outro, mas cada um continua do mesmo tamanho por dentro. Viu? Padding mexe por dentro, margin mexe na vizinhança.

✏️ Pratique

Cereja: F12 → inspecione um .consulta → no painel, o diagrama colorido do Box Model: conteúdo azul, padding verde, borda, margin laranja. O navegador literalmente desenha as quatro camadas para você. Essa imagem colorida é o mapa que você vai consultar pelos próximos anos.

🧠 Teste-se

Quero afastar a tabela do histórico do título dela: uso padding ou margin? E quero dar respiro interno ao botão de anexar exame: padding ou margin?

Ver resposta

Afastar do título = espaço entre vizinhos = margin. Respiro interno do botão = espaço entre conteúdo e borda = padding.

C6

Seletores básicos: elemento, classe e id

Boa notícia: você já usou os três seletores sem saber. Agora eles ganham nome e regra.

Seletor de elemento — p { color: gray; } — mira TODAS as tags daquele tipo. É o megafone na praça: "atenção, todos os parágrafos!". Foi o seu h1 e o seu body.

Seletor de classe — .destaque, com PONTO na frente — mira os elementos que carregam aquela class no HTML. É chamar por sobrenome: "família Silva, venham cá" — vários atendem. Foi o seu .lembrete e o .consulta. Reutilizável à vontade: dez elementos podem ter class="consulta".

Seletor de id — #cabecalho, com CERQUILHA — mira O elemento único que tem aquele id. É chamar por CPF: um, e só um, responde. Lembra da regra do HTML? Id não se repete na página — e o CSS respeita isso.

📌 Nota

Regra de bolso do mercado: classes para estilos reutilizáveis; ids apenas para elementos únicos e importantes — o cabeçalho, a navegação. Na prática do dia a dia, você vai estilizar 90% das coisas com classe.

✏️ Pratique

Um de cada tipo no Bem-cuidado, para sentir a diferença de alcance:

/* elemento: TODOS os h2 da página de uma vez */
h2 {
  color: #2c3e50;
}

/* classe: só quem tem class="logo" */
.logo {
  font-size: 24px;
  font-weight: 700;
}

/* id: unicamente a section de próximas consultas */
#proximas {
  background-color: #eef3f8;
  padding: 16px;
}
🧠 Teste-se

Quero deixar TODOS os links do menu sem sublinhado: elemento, classe ou id? E para destacar só a seção de anexos? E para criar um estilo "botao-perigo" que várias telas vão reusar?

Ver resposta

Todos os links → seletor de elemento (a). Só a seção de anexos → id (#anexos), pois ela é única. Estilo reutilizável → classe (.botao-perigo).

C7

Exemplo prático: a primeira identidade visual do Bem-cuidado

Antes de codar, leia este par de arquivos e tente EXPLICAR cada regra em voz alta (sério, funciona): o que o body ganha? (fonte Arial, fundo cinza-claro #f0f0f0.) O que a classe .titulo faz? (cor #333, 36px, centralizado — olha o text-align: center, propriedade nova!) E a .texto? (cor, tamanho, padding, margin e borda — o Box Model completo numa regra só.)

/* estilo.css do exemplo */
body {
  font-family: Arial, sans-serif;
  background-color: #f0f0f0;
}

.titulo {
  color: #333;
  font-size: 36px;
  text-align: center;
}

.texto {
  color: #666;
  font-size: 18px;
  padding: 20px;
  margin: 10px;
  border: 1px solid #ccc;
}

Se você conseguiu ler e explicar — parabéns: você está LENDO CSS. Algumas horas atrás isso era grego. Agora, o grande hands-on no projeto:

✏️ Pratique

Digite cada regra no seu styles.css, sempre salvando e olhando o resultado entre uma e outra:

/* topo do site com identidade */
.topo {
  background-color: #2c5f7c;
  color: white;
  padding: 16px;
}

.topo a {
  color: white;
}

/* saudação centralizada */
.saudacao {
  text-align: center;
  padding: 24px;
}

/* tabela do histórico mais legível */
table {
  background-color: white;
  border: 1px solid #dddddd;
  padding: 8px;
}

/* botões com cara de botão de verdade */
button {
  background-color: #2c5f7c;
  color: white;
  padding: 10px 18px;
  border: 1px solid #1e4258;
}

Dois detalhes novos de brinde. O .topo a — seletor composto: "os links QUE ESTÃO DENTRO de .topo" — é um gostinho de como seletores se combinam, tema das próximas aulas. E o padding: 10px 18px com dois valores: o primeiro é vertical (cima/baixo), o segundo horizontal (lados) — atalho que você vai usar todo dia.

Agora compare com a página de algumas horas atrás. Mesma estrutura, mesmo HTML — outra cara completamente. É a prova viva da separação: conteúdo de um lado, aparência do outro.

✏️ Pratique

Exercício livre: personalize UMA coisa do seu jeito — a paleta de cores do topo, o estilo do .lembrete, a borda dos cartões. Não existe resposta errada aqui; existe explorar.

C8

Boas práticas e a ferramenta secreta: F12

  • Use sempre arquivo externo — você já está fazendo: seu styles.css nasceu hoje. O style interno serve para teste rápido; o inline, quase nunca.
  • Nomeie classes com significado — .botao-primario, .cartao-produto: nomes que contam o que a coisa É. Nunca .classe1 ou .azul (e se o azul virar verde amanhã? O nome mente!). Repare que o Bem-cuidado já segue isso: .lembrete, .consulta, .saudacao — dá para entender o site só lendo as classes.
  • Teste no navegador constantemente — o ciclo salvou-olhou que você repetiu o guia inteiro. Nunca escreva 30 linhas de CSS às cegas para olhar só no final.
  • Comece simples e evolua — domine seletores, cores e box model antes de layouts complexos (exatamente o cardápio deste guia). Flexbox e Grid vêm na hora certa; ansiedade não acelera aprendizado.

E o terceiro item merece capítulo próprio, porque é a ferramenta mais poderosa para aprender CSS: o Inspecionar Elemento. Aperte F12 (ou botão direito → Inspecionar). Esse painel é o raio-X da página: você clica em qualquer elemento e vê TODO o CSS aplicado nele — inclusive de qualquer site do mundo.

✏️ Pratique

Laboratório F12 (5 min): (1) F12 na página do Bem-cuidado → ícone de setinha (canto do painel) → clique no h1 → veja a sua regra listada; (2) no painel Styles, mude a cor AO VIVO, arraste o font-size — a página reage em tempo real; (3) F5 → tudo volta ao normal.

O F12 é um rascunho mágico: experimente à vontade, nada é salvo. Achou o valor perfeito? Copie para o styles.css — aí sim vira permanente. E sim: dá para "inspecionar" qualquer site bonito da internet e aprender como fizeram. É estudo, não cola.

Desafio do trio (fechando o ciclo Git + CSS)

  • No repositório do trio, criar o styles.css e estilizar o Bem-cuidado — cada integrante em sua própria branch, estilizando a sua parte (mesma divisão do HTML: mídias / tabela e anexos / formulário).
  • Cada um abre um Pull Request; outro integrante revisa e faz o merge — três PRs, três revisões, uma main bonita.
  • Requisitos mínimos de CSS: uma paleta de cores combinada entre o trio, font-family definida no body, cartões .consulta com border/padding/margin, e pelo menos uma regra de cada tipo de seletor (elemento, classe e id).
  • Bônus: usar o F12 para "roubar" (estudar!) uma ideia de estilo de um site real e adaptar.
